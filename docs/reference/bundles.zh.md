# 套装

套装把现有的 Spec Kit 组件——扩展、预设、工作流和步骤——组合成一个带版本、可安装的整体。扩展和预设是基础组件（primitive），而套装是一个精选的组合栈：它声明团队或角色所需的一切，并通过每个组件自身的安装机制一步装齐。套装本身不增加任何新的运行时行为：它只是架在你已有基础组件之上的分发与组合层。

一个套装由 `bundle.yml` 清单描述，并通过与其他组件相同的目录源栈被发现。安装套装时，会按固定（pin）的版本解析其声明的组件，检查唯一的跨套装冲突点（当前生效的集成），然后以幂等方式应用每个组件，并记录完整的安装溯源信息，以便日后干净地移除或刷新。

## 搜索可用套装

```bash
specify bundle search [query]
```

| 选项 | 说明 |
| ----------- | ---------------------------- |
| `--offline` | 不访问网络 |
| `--json`    | 输出机器可读的 JSON |

在所有生效的目录源中搜索匹配查询的套装。不带查询时，列出每个可用套装及其版本、角色、来源和信任标识（组织策划的目录源条目为 `verified`，其余为 `community`），方便你在安装前判断可信度。

## 套装详情

```bash
specify bundle info <bundle_id>
```

| 选项 | 说明 |
| ------------ | --------------------------------- |
| `--offline`  | 不访问网络 |
| `--json`     | 输出机器可读的 JSON |

显示套装的完整元数据，以及它将安装的**完全展开的组件集**——每个扩展、预设、步骤和工作流及其固定的版本，外加预设的优先级与策略。输出同样包含信任标识（`verified` 或 `community`），方便安装前判断可信度。这份预览与 `install` 实际应用的计划完全一致，所以你在动手之前就能确切看到将添加什么。与已安装套装提供的组件之间可预见的重叠也会在这里显示出来。

## 安装套装

```bash
specify bundle install <bundle_id | path>
```

| 选项 | 说明 |
| ---------------- | ------------------------------------------------------------------ |
| `--integration`  | 覆盖初始化/安装时使用的集成 |
| `--offline`      | 不访问网络 |

通过每个基础组件自身的安装机制装入套装的完整组件集。参数可以是目录源中的套装 ID，也可以是本地路径——已构建的 `.zip` 构建产物、套装目录或 `bundle.yml` 文件；本地来源直接安装，不查询目录源栈。

如果当前目录还不是 Spec Kit 项目，`install` 会先初始化一个，让一份全新的检出（checkout）通过一条命令就达到可用状态。`--integration` 用于在初始化新项目时选择集成；当套装固定了特定集成、而项目当前生效的集成又无法确定（`.specify/integration.json` 缺失或不可读）时，它也用来确认目标集成。它**不会**覆盖已初始化项目的生效集成：如果套装指向的集成与项目的不同，安装会中止且不做任何更改。集成无关的套装会继承项目当前生效的集成。安装是幂等的——已经存在的组件会被跳过。安装失败时不会写入溯源记录（失败的安装什么也不记录），本次运行中已装入的组件会尽力（best-effort）移除——移除中的错误会被吞掉，因此磁盘上可能残留部分状态。

## 更新套装

```bash
specify bundle update [<bundle_id>]
```

| 选项 | 说明 |
| ---------------- | --------------------------------------------------------------------------------------------------------------------- |
| `--all`          | 更新所有已安装的套装 |
| `--integration`  | 覆盖刷新组件时使用的集成；仅在项目当前生效的集成无法确定时才生效 |
| `--offline`      | 不访问网络 |

重新解析套装并通过每个基础组件自身的更新路径**刷新**其组件，把已安装的组件升到套装最新固定的版本，同时保留组件层面的覆盖设置（例如预设优先级）。可以提供套装 ID，或用 `--all` 更新全部已安装套装。

> **版本固定只在安装时强制执行。** 幂等检查基于 ID，不感知版本：`install` 期间已存在的组件会被直接跳过，不会比较其磁盘上的版本与清单中固定的版本。因此，只有当套装机制首次实际安装某个组件或刷新它时，版本固定才有保证。运行 `specify bundle update` 可以把每个归属组件重新按固定版本应用一遍。

## 移除套装

```bash
specify bundle remove <bundle_id>
```

只卸载这个套装贡献的组件；其他已安装套装仍需要的组件会原样保留（不会连带移除）。

## 列出已安装的套装

```bash
specify bundle list
```

| 选项 | 说明 |
| -------- | ---------------------------- |
| `--json` | 输出机器可读的 JSON |

列出项目中已安装的套装，包括版本、组件数量和安装时间。

## 用套装初始化项目

```bash
specify bundle init [<bundle_id>]
```

| 选项 | 说明 |
| ---------------- | ---------------------------------------- |
| `--integration`  | 集成覆盖 |
| `--offline`      | 不访问网络 |

确保当前目录是一个 Spec Kit 项目（必要时幂等地初始化），然后可选地安装指定套装。适合作为新检出仓库的显式一步引导。

## 校验套装

```bash
specify bundle validate
```

| 选项 | 说明 |
| ------------ | ------------------------------------------------------------------- |
| `--path`     | 套装目录或 `bundle.yml`（默认：当前目录） |
| `--offline`  | 只对照套装内置组件和已安装组件校验引用 |

报告 `bundle.yml` 是否格式正确，以及声明的每个组件引用能否解析。引用会对照套装内置的组件、项目已安装的组件，以及——联网时——生效的目录源进行检查。只有当一个引用在所有可检查的位置都确定不存在时校验才会失败：也就是说，某个生效的目录源可达并确认该组件缺失。无法验证的引用——因为校验处于离线状态，或目录源不可达——会降级为警告而不是让整次运行失败，好让创作工作可以继续。

## 构建套装产物

```bash
specify bundle build
```

| 选项 | 说明 |
| ----------- | ------------------------------------------------------- |
| `--path`    | 套装目录（默认：当前目录） |
| `--output`  | 构建产物的输出目录 |

从套装目录产出一个带版本、可分发的 `.zip` 构建产物。产物内嵌清单，可以直接用 `specify bundle install <artifact.zip>` 安装。

## 发布套装

套装作者在本地校验和打包套装，然后把生成的构建产物和目录源元数据托管到用户能访问的位置。套装的目录源条目指向套装构建产物，但 `bundle.yml` 内声明的组件仍通过套装内置组件、已安装组件，或生效的扩展、预设、工作流和步骤目录源来解析。

如果你的套装引用了非默认目录源中的组件，请在文档中写明这些目录源 URL，并在添加了这些目录源的干净项目中测试安装路径。社区套装的提交应在 [Bundle Submission](https://github.com/github/spec-kit/issues/new?template=bundle_submission.yml) issue 中附上这份依赖解析证据。

## 管理目录源

套装通过一个按优先级排序的目录源栈（项目、用户和内置三个作用域）被发现。

### 列出目录源栈

```bash
specify bundle catalog list
```

打印当前生效、按优先级排序的目录源栈，以及每个来源的作用域和安装策略。

### 添加目录源

```bash
specify bundle catalog add <url>
```

| 选项 | 说明 |
| ------------- | ------------------------------------------------------- |
| `--policy`    | `install-allowed` 或 `discovery-only` |
| `--priority`  | 来源优先级（数字越小优先级越高；默认 10） |
| `--id`        | 显式指定来源 ID |

注册一个项目作用域的目录源并持久化。

### 移除目录源

```bash
specify bundle catalog remove <id_or_url>
```

移除一个项目作用域的目录源。内置的默认来源不能删除。

> **注意：** `search` 和 `info` 在任何位置都能运行——没有项目时回退到内置/用户目录源栈。其余会改变状态的命令（`list`、`update`、`remove`、`catalog`）要求项目已经用 `specify init` 初始化过。`install` 和 `init` 在未初始化的目录中运行时会按需初始化项目。
