# 处理复杂功能

大型或复杂的功能往往能顺利跑完 `/speckit.specify`、`/speckit.plan` 和 `/speckit.tasks`，却在实现阶段掉链子。在一次漫长的 `/speckit.implement` 运行中途，智能体可能开始跟丢计划、无视任务，甚至出现幻觉——通常就发生在上下文压缩被触发前后。

根本原因是上下文窗口耗尽。当一次实现运行试图把整个功能都装进上下文时，随着窗口被填满，模型的表现会持续劣化。解决办法是控制每次运行的范围，让它稳稳地待在上下文限制之内。

`/speckit.implement` 命令接受自由形式的用户输入，智能体在执行前必须先考虑这些输入。这意味着你无需改动任何工具，就能限定每次运行的范围。

## 方式 1：限制每次调用执行的任务数量

不要让 `/speckit.implement` 一口气跑完所有任务，而是让它提前收手：

```text
/speckit.implement 只执行任务 T001-T010，然后停下来汇报进度
```

或者按阶段限定范围：

```text
/speckit.implement 只执行 Setup 阶段，然后停止
```

由于已完成的任务会在 `tasks.md` 中标记为 `[X]`，下一次 `/speckit.implement` 调用会从你上次停下的地方继续。这让每次运行都稳稳地处于上下文限制之内。

## 方式 2：指示智能体使用子智能体

如果你的编码智能体支持子智能体（例如 GitHub Copilot CLI 或 VS Code 的 GitHub Copilot 扩展），可以指示 `/speckit.implement` 把单个任务委派出去：

```text
/speckit.implement 把每个并行 [P] 任务委派给一个子智能体
```

每个子智能体拿到的是聚焦的上下文——一个任务加上相关的计划节选——而不是完整的功能上下文，因此主会话中永远不会触发上下文压缩。

## 方式 3：两者结合

对于非常大的功能，把范围限定和任务委派结合起来：

```text
/speckit.implement 只执行 Core 阶段，把 [P] 任务委派给子智能体
```

## 方式 4：把功能拆解为更小的规范

当连单个阶段都会撑爆上下文时，把功能拆成可独立成规范的子功能。每个子功能拥有自己的 `spec.md`、`plan.md` 和 `tasks.md`，并各自走一遍 specify/plan/tasks/implement 循环。

这就是"规范的规范"（spec of specs）方式：第一轮迭代把一个庞大的功能拆解成更小、自成一体的规范，每个规范都能在不压垮模型的情况下完成实现。它的额外开销最大，所以只留给那些用其他任何办法都装不下的功能。

## 如何选择

| 方式 | 最适合 |
| --- | --- |
| 限制为 N 个任务或一个阶段 | 任何智能体；最简单；不需要子智能体支持 |
| 子智能体委派 | 支持子智能体的智能体；最大化并行度 |
| 范围限定 + 委派 | 在支持子智能体的智能体上做大型功能；两者兼顾 |
| 拆解为更小的规范 | 连单个阶段都会撑爆上下文时 |

大多数情况下，限制每次运行的任务范围是最简单的解法。当你的智能体支持子智能体且你想要并行时，再选子智能体委派；只有当单个阶段仍然大到一次跑不完时，才拆解为更小的规范。
