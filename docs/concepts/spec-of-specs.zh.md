# 规范的规范（Spec of Specs）

当一个功能大到无法在一次 `/speckit.specify` → `/speckit.plan` → `/speckit.tasks` → `/speckit.implement`
循环里跑完、而不至于让模型在实现中途跟丢时，你可以把它拆解成一份由更小、可独立成规范的子功能组成的
**路线图**。这就是"规范的规范"（spec of specs）方式：先用一轮前置梳理把一个庞大的功能拆解成自成一体的
规范，每个规范再各自走一遍 specify/plan/tasks/implement 循环。

> **何时动用它。** 在[处理复杂功能](complex-features.md)介绍的所有策略里，拆解带来的额外开销最大。
> **只有当那里更轻量的选项都不够用时**才动用它——先尝试限制每次 `/speckit.implement` 调用执行的任务
> 数量，再尝试子智能体委派，然后是两者结合。只有当连单个阶段都大到一次跑不完时，才动用规范的规范。

本页余下的部分讲的是*如何*用你手头已有的工具做到这一点。无需任何新命令或扩展。

## 路线图梳理

在编写任何子规范之前，先做一遍拆解梳理，产出一份路线图。把它当作与智能体的一次轻量规划对话，而不是
一份完整的规范：

1. **陈述整个功能。** 用一两句话描述这个大型功能（即"史诗"，epic），让智能体一开始就掌握全貌。
2. **识别可独立的切片。** 让智能体提出一小组子功能，每个都交付史诗中一块自洽的功能，并且能独立
   成规范。目标是让切片可独立测试——哪怕只实现其中一个，也应该能得到可演示的东西。
3. **划清边界。** 为每个切片写一句意图，并明确一条范围边界（哪些纳入，哪些推迟给兄弟切片）。正是
   清晰的边界，才让每份子规范小到能塞进上下文。
4. **按依赖排序。** 标注哪些切片依赖其他切片，并安排顺序，让前置条件先行。彼此之间没有依赖的切片
   可以按任意顺序构建。若要并行构建相互独立的切片，请使用独立的 worktree，让每次运行都拥有隔离的
   活跃功能状态。
5. **把结果记录为一份路线图。** 把这些切片沉淀到一个持久的路线图文件里（见下文），让后续每份子规范
   都能回指到它。

路线图刻意保持浅层：它只为子功能命名和排序，**不**做设计。设计发生在每个切片各自走一遍
`/speckit.specify` 的时候。

## 路线图产物

路线图就是一个你亲手编写、纳入版本控制的普通 Markdown 文件——背后没有任何特殊工具。把它放在子规范
能找到的地方：

- 对于功能范围内的史诗：`specs/<epic-slug>/roadmap.md`。
- 对于更大、横切多处的史诗：放在顶层的 `ROADMAP.md`。

路线图的每个条目都带有一个稳定的 id（后续用于建立链接）、一个名称、它的意图、它的范围边界、它的依赖、
一个状态，以及——一旦子规范存在——一个指向它的链接。一个最简模板：

```markdown
# 路线图：<史诗名称>

<一两句话：这个史诗是什么，以及为什么要拆解它。>

**状态图例**：planned · in-progress · done

| ID | Sub-feature | Intent | Scope boundary | Depends on | Status | Sub-spec |
|----|-------------|--------|----------------|-----------|--------|----------|
| R1 | <名称> | <一句话> | <纳入 / 推迟> | — | planned | — |
| R2 | <名称> | <一句话> | <纳入 / 推迟> | R1 | planned | — |
| R3 | <名称> | <一句话> | <纳入 / 推迟> | R1 | planned | — |
```

一旦某份子规范引用了 `ID` 列，就保持它不变——它是可追溯性的锚点。在创建每个子功能的规范目录时，把
路径填进 `Sub-spec` 列，并随着工作推进更新 `Status`。

## 为每个子功能编写规范

有了路线图，就用常规的 Spec Kit 流程逐条处理这些条目——没有新东西要学：

1. 挑选下一个依赖已经 `done`（或没有依赖）的路线图条目。
2. 只针对该切片运行 `/speckit.specify`，仅描述路线图条目里它的意图和范围。因为切片是有边界的，它的
   规范、计划和任务清单都稳稳地待在上下文窗口之内。
3. 像往常一样为该切片运行 `/speckit.plan`、`/speckit.tasks` 和 `/speckit.implement`。
4. 把该路线图条目标记为 `done`，然后处理下一个。

每个切片都是一个完整、独立的 Spec Kit 功能，拥有自己的 `spec.md`/`plan.md`/`tasks.md`。而路线图正是
把它们串联起来的东西。

## 把子规范链接到路线图

为了防止范围和意图在各次独立运行之间漂移，每份子规范都引用它的路线图条目，而路线图也反向链接回来——
这是一个简单、可用 grep 检索、双向的约定：

- **子规范 → 路线图。** 在子功能的 `spec.md` 里，在 `输入` / 摘要行中写明父路线图和条目 id，例如：

  ```markdown
  **输入**：父路线图：`specs/<epic>/roadmap.md` → 条目 **R3**。<功能描述>
  ```

- **路线图 → 子规范。** 在路线图表格里，把该条目的 `Sub-spec` 列设为子功能的目录，例如
  `specs/<epic>-part-3/`。

因为两个方向都是纯文本，你只需快速搜索就能把任何子规范追溯回它在史诗中的位置（并找到它的兄弟规范）——
不需要工具，也不需要元数据 schema。

## 让路线图与子规范保持同步

路线图是一份活文档。随着你了解得更多，让它和子规范保持一致：

- **先改路线图，再对账。** 当范围发生变化时，先更新路线图条目，再更新它影响到的所有子规范。路线图
  是史诗如何划分的事实来源。
- **尊重依赖与顺序。** 如果一个切片依赖另一个，先构建前置切片，并在依赖它的子规范里交叉引用，让这
  层关系从两边都能看见。
- **切片仍然太大时就递归。** 如果某个子功能大到无法在一个循环里成规范，就给它自己一份路线图，进一步
  拆解——同一套办法往下再套一层即可。递归会带来额外开销，所以只下探到上下文问题真正需要的深度。

## 完整示例

假设史诗是**"添加一个自助计费门户"**——大得远远超出一个循环的处理能力。路线图梳理把它拆成三个可独立
成规范的切片。

`specs/billing-portal/roadmap.md`：

```markdown
# 路线图：自助计费门户

让客户无需联系客服即可查看发票、管理支付方式并变更套餐。它大到一个循环装不下，因此被拆分为多个独立
的切片。

**状态图例**：planned · in-progress · done

| ID | Sub-feature | Intent | Scope boundary | Depends on | Status | Sub-spec |
|----|-------------|--------|----------------|-----------|--------|----------|
| R1 | 发票历史 | 客户查看并下载历史发票 | 只读；无支付操作 | — | done | specs/billing-invoices/ |
| R2 | 支付方式 | 添加、移除并设置默认卡片 | 不涉及套餐变更；假定发票功能已存在 | R1 | in-progress | specs/billing-payment-methods/ |
| R3 | 套餐变更 | 升级/降级订阅套餐 | 使用 R2 的默认支付方式 | R1, R2 | planned | — |
```

随后各个切片再各自成规范。例如，**R2** 这个子功能的 `spec.md` 开头带一条反向引用：

```markdown
# 功能规范：计费 — 支付方式

**输入**：父路线图：`specs/billing-portal/roadmap.md` → 条目 **R2**。
让客户在计费门户中添加、移除并设置默认支付方式。
```

从这里，读者就能把 **R2** 追溯回路线图，看到它依赖 **R1**（发票历史，已经 `done`），也看到 **R3**
（套餐变更）正等着它。先构建 R1，再 R2，再 R3，能让每次运行都保持小巧，而路线图则守住整个史诗的形态。

## 面向自动化（可选）

如果你更愿意自动完成路线图的记录与一致性检查，而不是手工维护这个文件，社区维护的
[Spec Roadmap 扩展](https://github.com/srobroek/speckit-roadmap)正朝这个方向探索。它是第三方扩展，
并非必需——上面的手工约定本身就已经够用。
