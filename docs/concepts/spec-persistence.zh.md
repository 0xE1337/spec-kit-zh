# 规范留存模型

Spec Kit 有意把"需求变化后 `spec.md`、`plan.md` 和 `tasks.md` 何去何从"的决定权留给团队。工具包提供的是一套可重复的工作流，但不强制任何一种产物维护策略。

本页为三种常见模型命名，方便团队把这个选择摆到明面上。它们没有哪个是默认项，Spec Kit 也不要求使用其中任何一个。

## 两个彼此独立的问题

规范驱动开发有一个时间维度的问题：规范应该"有效"多久？一篇 [SDD 工具综述](https://martinfowler.com/articles/exploring-gen-ai/sdd-3-tools.html)把这个生命周期分成三个层次：

- **规范先行（Spec-first）**：编码前先写规范，之后允许丢弃。
- **规范锚定（Spec-anchored）**：实现完成后保留规范，用于后续变更。
- **规范即源（Spec-as-source）**：把规范当作唯一由人编辑的源头，实现产物都从它再生成。

Spec Kit 还揭示了第二个问题：当需求变化时，整套产物会发生什么？下面的模型描述的就是这种变更策略。

## 回流式规范（Flow-Back Spec）

当 `spec.md`、`plan.md`、`tasks.md` 与实现被允许互相影响时，使用回流式。

在这个模型中，编辑可以从任何产物开始。开发者可能在实现过程中更新 `tasks.md`，在技术上有新发现后修订 `plan.md`，或在产品澄清后调整 `spec.md`。之后团队手工核对整套产物，让最终的项目历史仍然讲得通。

回流式适合以下情况：

- 团队足够小，能及时发现并调和产物漂移
- 预期实现中的发现会重塑最初的计划
- 速度比把每个中间决策保留为不可变历史更重要

主要风险是悄无声息的分歧。如果团队修改了低层产物却没有把决策回写到 `spec.md`，后来的贡献者可能不知道该信任哪个产物。

## 前推式规范（Flow-Forward Spec）

当每个功能目录都应保持为历史记录时，使用前推式。

在这个模型中，已完成的产物被视为不可变。需求变化时，团队创建一个新的功能目录，而不是修改既有的 `spec.md`、`plan.md` 或 `tasks.md`。旧目录仍然有用——用于审计、对比，或解释项目如何走到今天的状态。

前推式适合以下情况：

- 可审计性和可追溯性很重要
- 功能边界清晰，很少原地返工
- 团队希望有一条清晰的需求变更时间线

主要代价是重复。相关决策可能分散在多个功能目录中，因此团队需要通过命名、链接或评审习惯，让脉络容易追踪。

## 活规范（Living Spec）

当 `spec.md` 是契约、其他产物都由它派生时，使用活规范。

在这个模型中，团队先更新 `spec.md`，再从这个源头重新生成或修订 `plan.md` 和 `tasks.md`。计划和任务清单仍然有价值，但它们被当作可抛弃的派生物，而不是永久的事实来源。

活规范适合以下情况：

- 产品契约足够稳定，能够主导工作流
- 团队能接受在规范变化后重新生成派生产物
- 需求与实现的一致性比保留每份中间计划更重要

主要风险是：如果丢弃派生产物时没有把重要决策保存到别处，有价值的实现理由可能随之丢失。

## 选择模型

模型是团队约定，不是 CLI 设置。一个项目甚至可以在不同区域使用不同模型，只要贡献者清楚各处适用哪一种。

| 模型 | 变更规则 | 最适合 | 需要提防 |
|---|---|---|---|
| 回流式规范 | 任意产物可编辑，随后再核对 | 快速迭代与紧密协作 | 产物之间悄然漂移 |
| 前推式规范 | 新需求创建新的功能目录 | 审计线索与历史清晰度 | 上下文重复或碎片化 |
| 活规范 | 只编辑 `spec.md`；重新生成派生产物 | 规范即契约 | 再生成的文件中丢失决策理由 |

如果你的团队还没有选定模型，先回答两个问题：

1. 已完成的功能目录应该是历史记录，还是可编辑的工作区？
2. `spec.md` 是唯一事实来源，还是允许 `plan.md` 和 `tasks.md` 成为平级的事实来源？

答案明确后，把这条约定写进项目宪章或团队入职说明，让后来的贡献者知道该如何处理变更。
