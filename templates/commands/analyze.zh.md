## 执行前检查

**检查扩展钩子（分析之前）**：
- 检查项目根目录是否存在 `.specify/extensions.yml`。
- 如果存在，读取它并查找 `hooks.before_analyze` 键下的条目
- 如果 YAML 无法解析或格式无效，静默跳过钩子检查并正常继续
- 过滤掉 `enabled` 显式为 `false` 的钩子。没有 `enabled` 字段的钩子默认视为启用。
- 对于剩余的每个钩子，**不要**尝试解释或求值钩子的 `condition` 表达式：
  - 如果钩子没有 `condition` 字段，或该字段为 null/空，将钩子视为可执行
  - 如果钩子定义了非空的 `condition`，跳过该钩子，把条件求值留给 HookExecutor 实现
- 对于每个可执行的钩子，根据其 `optional` 标志输出以下内容：
  - **可选钩子**（`optional: true`）：
    ```
    ## 扩展钩子

    **可选前置钩子**：{extension}
    命令：`/{command}`
    说明：{description}

    提示词：{prompt}
    执行方式：`/{command}`
    ```
  - **强制钩子**（`optional: false`）：
    ```
    ## 扩展钩子

    **自动前置钩子**：{extension}
    正在执行：`/{command}`
    EXECUTE_COMMAND: {command}

    等待钩子命令的结果返回后，再继续进入"目标"。
    ```
    输出上面的块之后，你**必须**实际调用该钩子，并等待其完成后再继续。以你在当前智能体/会话中自己运行该命令的方式来运行它（实际调用形式可能与上面显示的字面 `{command}` id 不同，例如技能模式的智能体会以 `/skill:speckit-...` 或 `$speckit-...` 的形式运行它）。仅输出该块并不等于运行了钩子。
- 如果没有注册任何钩子，或 `.specify/extensions.yml` 不存在，则静默跳过

## 目标

在实现之前，识别三个核心产物（`spec.md`、`plan.md`、`tasks.md`）之间的不一致、重复、模糊和描述不足之处。本命令必须在 `__SPECKIT_COMMAND_TASKS__` 成功产出完整的 `tasks.md` 之后才能运行。

## 运行约束

**严格只读**：**不要**修改任何文件。输出一份结构化的分析报告。可以提供一份可选的整改方案（必须由用户明确批准后，才会手动调用任何后续的编辑命令）。

**宪章权威性**：项目宪章（`/memory/constitution.md`）在本分析范围内**不可协商**。与宪章的冲突自动视为 CRITICAL，需要通过调整规范、计划或任务来解决——不允许稀释、重新解释或悄悄忽略原则。如果原则本身需要修改，必须在 `__SPECKIT_COMMAND_ANALYZE__` 之外通过一次单独、明确的宪章更新来完成。

## 执行步骤

### 1. 初始化分析上下文

在仓库根目录运行一次 `{SCRIPT}`，解析 JSON 中的 FEATURE_DIR 和 AVAILABLE_DOCS。推导绝对路径：

- SPEC = FEATURE_DIR/spec.md
- PLAN = FEATURE_DIR/plan.md
- TASKS = FEATURE_DIR/tasks.md

如果任何必需文件缺失，中止并给出错误信息（指示用户运行缺失的前置命令）。
参数中包含单引号时（如 "I'm Groot"），使用转义语法：例如 'I'\''m Groot'（或在可能时改用双引号："I'm Groot"）。

### 2. 加载产物（渐进式披露）

只从每个产物中加载最少必要的上下文：

**从 spec.md：**

- 概述/上下文
- 功能需求
- 成功标准（可度量的结果——例如性能、安全性、可用性、用户成功、业务影响）
- 用户故事
- 边界情况（如有）

**从 plan.md：**

- 架构/技术栈选择
- 数据模型引用
- 阶段
- 技术约束

**从 tasks.md：**

- 任务 ID
- 描述
- 阶段分组
- 并行标记 [P]
- 引用的文件路径

**从宪章：**

- 加载 `/memory/constitution.md` 用于原则校验

### 3. 构建语义模型

创建内部表示（不要在输出中包含原始产物）：

- **需求清单**：为每条功能需求（FR-###）和成功标准（SC-###）记录一个稳定键。存在显式的 FR-/SC- 标识符时以其为主键，也可以另外派生一个祈使短语式的 slug 以便阅读（例如 "User can upload file" → `user-can-upload-file`）。只纳入需要实际构建工作的成功标准项（例如负载测试基础设施、安全审计工具），排除上线后的结果指标和业务 KPI（例如"将支持工单减少 50%"）。
- **用户故事/动作清单**：带验收标准的离散用户动作
- **任务覆盖映射**：把每个任务映射到一条或多条需求或故事（通过关键词推断，或 ID、关键短语等显式引用模式）
- **宪章规则集**：提取原则名称和 MUST/SHOULD 规范性声明

### 4. 检测轮次（节省 token 的分析）

聚焦高信号的发现。发现总数上限为 50 条；其余在溢出摘要中汇总。

#### A. 重复检测

- 识别近似重复的需求
- 标记表述质量较低的一方以便合并

#### B. 模糊性检测

- 标记缺乏可度量标准的模糊形容词（快速、可扩展、安全、直观、健壮）
- 标记未解决的占位符（TODO、TKTK、???、`<placeholder>` 等）

#### C. 描述不足

- 有动词但缺少宾语或可度量结果的需求
- 与验收标准缺乏对应关系的用户故事
- 引用了规范/计划中未定义的文件或组件的任务

#### D. 宪章对齐

- 与某条 MUST 原则冲突的任何需求或计划元素
- 缺少宪章要求的章节或质量关卡

#### E. 覆盖缺口

- 没有任何关联任务的需求
- 没有映射到需求/故事的任务
- 需要实际构建工作（性能、安全性、可用性）却未体现在任务中的成功标准

#### F. 不一致

- 术语漂移（同一概念在不同文件中叫法不同）
- 计划中引用但规范中不存在的数据实体（或反之）
- 任务排序矛盾（例如集成任务排在基础准备任务之前且没有依赖说明）
- 相互冲突的需求（例如一条要求 Next.js，另一条指定 Vue）

### 5. 严重级别判定

使用以下启发式规则确定发现的优先级：

- **CRITICAL**：违反宪章 MUST 原则、缺失核心规范产物，或某条需求零覆盖且阻塞基线功能
- **HIGH**：重复或相互冲突的需求、模糊的安全/性能属性、无法测试的验收标准
- **MEDIUM**：术语漂移、缺少非功能性任务覆盖、描述不足的边界情况
- **LOW**：风格/措辞改进、不影响执行顺序的轻微冗余

输出一份 Markdown 报告（不写入文件），结构如下：

## 运行原则

### 上下文效率

- **最少的高信号 token**：聚焦可执行的发现，而不是详尽无遗的记录
- **渐进式披露**：增量加载产物；不要把全部内容一次性倒进分析
- **节省 token 的输出**：发现表最多 50 行；溢出部分做汇总
- **结果确定性**：在没有变更的情况下重新运行，应产生一致的 ID 和计数

### 分析准则

- **绝不修改文件**（这是只读分析）
- **绝不虚构缺失的章节**（如果不存在，如实报告）
- **优先处理宪章违规**（这些始终是 CRITICAL）
- **用示例代替详尽的规则**（引用具体实例，而不是泛化的模式）
- **零问题时也要得体地报告**（输出带覆盖统计的成功报告）
