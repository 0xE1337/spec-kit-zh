## 检查清单的目的："给英语写单元测试"

**关键概念**：检查清单是**针对需求写作的单元测试**——它验证的是给定领域内需求的质量、清晰度和完整性。

**不是用于验证/测试**：

- ❌ 不是"验证按钮能正确点击"
- ❌ 不是"测试错误处理是否工作"
- ❌ 不是"确认 API 返回 200"
- ❌ 不是检查代码/实现是否符合规范

**而是用于需求质量验证**：

- ✅ "是否为所有卡片类型定义了视觉层级需求？"（完整性）
- ✅ "'醒目展示'是否用具体的尺寸/位置量化了？"（清晰度）
- ✅ "悬停状态需求在所有可交互元素间是否一致？"（一致性）
- ✅ "是否为键盘导航定义了无障碍需求？"（覆盖度）
- ✅ "规范是否定义了 logo 图片加载失败时会发生什么？"（边界情况）

**比喻**：如果说你的规范是用英语写的代码，检查清单就是它的单元测试套件。你测试的是需求写得好不好——完整、无歧义、可以进入实现——而**不是**实现能不能工作。

## 用户输入

```text
$ARGUMENTS
```

在继续之前，你**必须**考虑用户输入（如果不为空）。

## 执行前检查

**检查扩展钩子（生成检查清单之前）**：
- 检查项目根目录是否存在 `.specify/extensions.yml`。
- 如果存在，读取它并查找 `hooks.before_checklist` 键下的条目
- 如果 YAML 无法解析或格式无效，静默跳过钩子检查并正常继续
- 过滤掉 `enabled` 显式为 `false` 的钩子。没有 `enabled` 字段的钩子默认视为启用。
- 对其余每个钩子，**不要**尝试解释或求值钩子的 `condition` 表达式：
  - 如果钩子没有 `condition` 字段，或该字段为 null/空，将钩子视为可执行
  - 如果钩子定义了非空的 `condition`，跳过该钩子，把条件求值留给 HookExecutor 实现
- 对每个可执行的钩子，根据其 `optional` 标志输出以下内容：
  - **可选钩子**（`optional: true`）：
    ```
    ## 扩展钩子

    **可选前置钩子**：{extension}
    命令：`/{command}`
    描述：{description}

    提示词：{prompt}
    执行方式：`/{command}`
    ```
  - **强制钩子**（`optional: false`）：
    ```
    ## 扩展钩子

    **自动前置钩子**：{extension}
    正在执行：`/{command}`
    EXECUTE_COMMAND: {command}

    等待钩子命令的结果后，再继续执行"执行步骤"部分。
    ```
    输出上面的代码块之后，你必须真正调用该钩子，并等待它完成后再继续。请像你在本智能体/会话中亲自运行该命令那样运行它（实际调用方式可能与上面所示的字面 `{command}` id 不同，例如技能模式的智能体会以 `/skill:speckit-...` 或 `$speckit-...` 的形式运行）。仅输出代码块本身并不会运行钩子。
- 如果没有注册任何钩子，或 `.specify/extensions.yml` 不存在，静默跳过

## 执行步骤

1. **准备**：在仓库根目录运行 `{SCRIPT}`，解析 JSON 中的 FEATURE_DIR 和 AVAILABLE_DOCS 列表。
   - 所有文件路径必须是绝对路径。
   - 参数中含单引号（如 "I'm Groot"）时，使用转义语法：例如 'I'\''m Groot'（或尽可能用双引号："I'm Groot"）。

2. **如果存在**：加载 `/memory/constitution.md`，获取项目原则和治理约束。

3. **澄清意图（动态）**：动态推导最多三个初始的上下文澄清问题（没有预制的问题目录）。这些问题必须：
   - 由用户的措辞 + 从 spec/plan/tasks 中提取的信号生成
   - 只询问会实质改变检查清单内容的信息
   - 如果某一项在 `$ARGUMENTS` 中已经明确，就单独跳过该项
   - 精确优先于宽泛

   生成算法：
   1. 提取信号：功能领域关键词（例如 auth、latency、UX、API）、风险指示词（"critical"、"must"、"compliance"）、相关方线索（"QA"、"review"、"security team"），以及明确的交付物（"a11y"、"rollback"、"contracts"）。
   2. 把信号聚类为候选关注领域（最多 4 个），按相关性排序。
   3. 如果未明确说明，推断可能的受众与使用时机（作者、评审者、QA、发布）。
   4. 检测缺失的维度：范围广度、深度/严格程度、风险侧重、排除边界、可度量的验收标准。
   5. 从以下原型中选择并组织问题：
      - 范围细化（例如"这份清单应该包含与 X 和 Y 的集成触点，还是只限于本模块的正确性？"）
      - 风险优先级（例如"这些潜在风险领域中，哪些应该设置强制的关卡检查？"）
      - 深度校准（例如"这是一份轻量的提交前健全性清单，还是正式的发布关卡？"）
      - 受众定位（例如"这份清单只给作者自己用，还是同事在 PR 评审时也会用？"）
      - 边界排除（例如"这一轮是否应该明确排除性能调优类条目？"）
      - 场景类别缺口（例如"没有检测到恢复流程——回滚/部分失败路径在范围内吗？"）

   问题格式规则：
   - 如果呈现选项，生成一个紧凑的表格，列为：选项 | 候选 | 为何重要
   - 最多 A-E 五个选项；如果自由作答更清晰，就省略表格
   - 绝不要求用户复述他们已经说过的内容
   - 避免猜测性的类别（不要幻觉）。不确定时，明确询问："请确认 X 是否属于范围。"

   无法交互时的默认值：
   - 深度：标准
   - 受众：与代码相关时为评审者（PR）；否则为作者
   - 关注领域：相关性最高的前 2 个聚类

   输出这些问题（标记为 Q1/Q2/Q3）。收到答案后：如果仍有 ≥2 个场景类别（备选/异常/恢复/非功能领域）不明确，你可以再追加最多两个针对性的后续问题（Q4/Q5），每个附一行理由（例如"恢复路径风险未解决"）。总共不得超过 5 个问题。如果用户明确拒绝更多问题，就不再追加。

4. **理解用户请求**：综合 `$ARGUMENTS` + 澄清答案：
   - 推导检查清单主题（例如 security、review、deploy、ux）
   - 汇总用户明确提出的必备条目
   - 把关注领域的选择映射到类别脚手架
   - 从 spec/plan/tasks 推断缺失的上下文（不要幻觉）

5. **加载功能上下文**：从 FEATURE_DIR 读取：
   - spec.md：功能需求与范围
   - plan.md（如果存在）：技术细节、依赖
   - tasks.md（如果存在）：实现任务

   **上下文加载策略**：
   - 只加载与当前关注领域相关的必要部分（避免整个文件倾倒进来）
   - 优先把长章节总结为简明的场景/需求条目
   - 采用渐进式披露：只有在发现缺口时才追加检索
   - 如果源文档很大，生成中间摘要条目，而不是嵌入原始文本

6. **生成检查清单**——创建"针对需求的单元测试"：
   - 如果 `FEATURE_DIR/checklists/` 目录不存在则创建
   - 生成唯一的检查清单文件名：
     - 使用基于领域的简短描述性名称（例如 `ux.md`、`api.md`、`security.md`）
     - 格式：`[domain].md`
   - 文件处理行为：
     - 如果文件不存在：创建新文件，条目编号从 CHK001 开始
     - 如果文件已存在：把新条目追加到现有文件，编号接续最后一个 CHK ID（例如最后一条是 CHK015，则新条目从 CHK016 开始）
   - 绝不删除或替换已有的检查清单内容——始终保留并追加

   **核心原则——测试需求，而不是测试实现**：
   每个检查清单条目都必须评估需求本身的：
   - **完整性**：所有必要的需求都在吗？
   - **清晰度**：需求是否无歧义且具体？
   - **一致性**：需求之间是否相互一致？
   - **可度量性**：需求能否被客观验证？
   - **覆盖度**：所有场景/边界情况都覆盖了吗？

   **类别结构**——按需求质量维度给条目分组：
   - **需求完整性**（所有必要的需求都记录了吗？）
   - **需求清晰度**（需求是否具体且无歧义？）
   - **需求一致性**（需求之间是否对齐且没有冲突？）
   - **验收标准质量**（成功标准可度量吗？）
   - **场景覆盖**（所有流程/情况都覆盖了吗？）
   - **边界情况覆盖**（边界条件定义了吗？）
   - **非功能需求**（性能、安全、无障碍等——是否已明确？）
   - **依赖与假设**（是否已记录并验证？）
   - **歧义与冲突**（哪些地方需要澄清？）

   **如何撰写检查清单条目——"给英语写单元测试"**：

   ❌ **错误**（测试实现）：
   - "验证落地页显示 3 张剧集卡片"
   - "测试桌面端悬停状态是否工作"
   - "确认点击 logo 能跳转到首页"

   ✅ **正确**（测试需求质量）：
   - "精选剧集的确切数量和布局是否已明确规定？"[Completeness]
   - "'醒目展示'是否用具体的尺寸/位置量化了？"[Clarity]
   - "悬停状态需求在所有可交互元素间是否一致？"[Consistency]
   - "是否为所有可交互 UI 定义了键盘导航需求？"[Coverage]
   - "logo 图片加载失败时的回退行为是否已明确？"[Edge Cases]
   - "是否为异步加载的剧集数据定义了加载状态？"[Completeness]
   - "规范是否为相互竞争的 UI 元素定义了视觉层级？"[Clarity]

   **条目结构**：
   每个条目应遵循以下模式：
   - 用问句形式询问需求质量
   - 聚焦规范/计划里写了什么（或没写什么）
   - 用方括号标注质量维度 [Completeness/Clarity/Consistency 等]
   - 检查既有需求时引用规范章节 `[Spec §X.Y]`
   - 检查缺失需求时使用 `[Gap]` 标记

   **按质量维度分类的示例**：

   完整性：
   - "是否为所有 API 失败模式定义了错误处理需求？[Gap]"
   - "是否为所有可交互元素明确了无障碍需求？[Completeness]"
   - "是否为响应式布局定义了移动端断点需求？[Gap]"

   清晰度：
   - "'快速加载'是否用具体的时间阈值量化了？[Clarity, Spec §NFR-2]"
   - "'相关剧集'的选取标准是否已明确定义？[Clarity, Spec §FR-5]"
   - "'醒目'是否用可度量的视觉属性定义了？[Ambiguity, Spec §FR-4]"

   一致性：
   - "导航需求在所有页面间是否一致？[Consistency, Spec §FR-10]"
   - "卡片组件需求在落地页和详情页之间是否一致？[Consistency]"

   覆盖度：
   - "是否为零状态场景（没有剧集）定义了需求？[Coverage, Edge Case]"
   - "是否覆盖了并发用户交互场景？[Coverage, Gap]"
   - "是否为部分数据加载失败明确了需求？[Coverage, Exception Flow]"

   可度量性：
   - "视觉层级需求是否可度量/可测试？[Acceptance Criteria, Spec §FR-1]"
   - "'均衡的视觉权重'能否被客观验证？[Measurability, Spec §FR-2]"

   **场景分类与覆盖**（聚焦需求质量）：
   - 检查是否为以下场景类别定义了需求：主要（Primary）、备选（Alternate）、异常/错误（Exception/Error）、恢复（Recovery）、非功能（Non-Functional）
   - 对每个场景类别，追问："[场景类型] 的需求是否完整、清晰且一致？"
   - 如果某个场景类别缺失："[场景类型] 需求是有意排除还是遗漏了？[Gap]"
   - 涉及状态变更时纳入韧性/回滚："是否为迁移失败定义了回滚需求？[Gap]"

   **可追溯性要求**：
   - 最低要求：≥80% 的条目必须包含至少一个可追溯引用
   - 每个条目应引用：规范章节 `[Spec §X.Y]`，或使用这些标记：`[Gap]`、`[Ambiguity]`、`[Conflict]`、`[Assumption]`
   - 如果不存在 ID 体系："是否建立了需求与验收标准的 ID 编号方案？[Traceability]"

   **暴露并解决问题**（需求质量问题）：
   针对需求本身提问：
   - 歧义："'快'这个词是否用具体指标量化了？[Ambiguity, Spec §NFR-1]"
   - 冲突："§FR-10 和 §FR-10a 之间的导航需求是否冲突？[Conflict]"
   - 假设："'播客 API 始终可用'这一假设是否经过验证？[Assumption]"
   - 依赖："外部播客 API 的需求是否已记录？[Dependency, Gap]"
   - 缺失的定义："'视觉层级'是否用可度量的标准定义了？[Gap]"

   **内容整合**：
   - 软上限：如果原始候选条目超过 40 个，按风险/影响排出优先级
   - 合并检查同一需求侧面的近似重复项
   - 如果低影响的边界情况超过 5 个，合并为一条："需求中是否覆盖了边界情况 X、Y、Z？[Coverage]"

   **🚫 绝对禁止**——以下写法会让它变成实现测试，而不是需求测试：
   - ❌ 任何以"验证"、"测试"、"确认"、"检查"+ 实现行为开头的条目
   - ❌ 引用代码执行、用户操作、系统行为
   - ❌ "正确显示"、"正常工作"、"按预期运行"
   - ❌ "点击"、"跳转"、"渲染"、"加载"、"执行"
   - ❌ 测试用例、测试计划、QA 流程
   - ❌ 实现细节（框架、API、算法）

   **✅ 必须使用的模式**——以下写法才是在测试需求质量：
   - ✅ "是否为 [场景] 定义/明确/记录了 [需求类型]？"
   - ✅ "[模糊用语] 是否用具体标准量化/澄清了？"
   - ✅ "[章节 A] 和 [章节 B] 之间的需求是否一致？"
   - ✅ "[需求] 能否被客观度量/验证？"
   - ✅ "需求中是否覆盖了 [边界情况/场景]？"
   - ✅ "规范是否定义了 [缺失的方面]？"

7. **结构参考**：按照 `templates/checklist-template.md` 中的规范模板生成检查清单，遵循其标题、元信息区、类别标题和 ID 格式。如果模板不可用，则使用：H1 标题、目的/创建时间元信息行、`##` 类别小节，其中包含 `- [ ] CHK### <需求条目>` 格式的行，ID 从 CHK001 开始全局递增。

8. **报告**：输出检查清单文件的完整路径、条目数量，并说明本次运行是新建了文件还是追加到了现有文件。总结：
   - 选定的关注领域
   - 深度级别
   - 参与者/时机
   - 已纳入的、用户明确指定的必备条目

**重要**：每次调用 `__SPECKIT_COMMAND_CHECKLIST__` 命令都使用一个简短的描述性检查清单文件名，并且要么新建文件，要么追加到现有文件。这样可以：

- 支持多份不同类型的检查清单（例如 `ux.md`、`test.md`、`security.md`）
- 使用简单好记、能表明清单用途的文件名
- 便于在 `checklists/` 文件夹中识别和浏览

为避免杂乱，请使用描述性的类型名，并在用完后清理过时的检查清单。

## 检查清单类型与条目示例

**UX 需求质量：** `ux.md`

条目示例（测试需求，而不是测试实现）：

- "视觉层级需求是否用可度量的标准定义了？[Clarity, Spec §FR-1]"
- "UI 元素的数量和位置是否已明确规定？[Completeness, Spec §FR-1]"
- "交互状态需求（悬停、聚焦、激活）是否定义得前后一致？[Consistency]"
- "是否为所有可交互元素明确了无障碍需求？[Coverage, Gap]"
- "图片加载失败时的回退行为是否已定义？[Edge Case, Gap]"
- "'醒目展示'能否被客观度量？[Measurability, Spec §FR-4]"

**API 需求质量：** `api.md`

条目示例：

- "是否为所有失败场景明确了错误响应格式？[Completeness]"
- "速率限制需求是否用具体阈值量化了？[Clarity]"
- "认证需求在所有端点间是否一致？[Consistency]"
- "是否为外部依赖定义了重试/超时需求？[Coverage, Gap]"
- "版本策略是否已写入需求文档？[Gap]"

**性能需求质量：** `performance.md`

条目示例：

- "性能需求是否用具体指标量化了？[Clarity]"
- "是否为所有关键用户旅程定义了性能目标？[Coverage]"
- "是否明确了不同负载条件下的性能需求？[Completeness]"
- "性能需求能否被客观度量？[Measurability]"
- "是否为高负载场景定义了降级需求？[Edge Case, Gap]"

**安全需求质量：** `security.md`

条目示例：

- "是否为所有受保护资源明确了认证需求？[Coverage]"
- "是否为敏感信息定义了数据保护需求？[Completeness]"
- "威胁模型是否已记录、需求是否与之对齐？[Traceability]"
- "安全需求与合规义务是否一致？[Consistency]"
- "是否定义了安全故障/泄露的响应需求？[Gap, Exception Flow]"

## 反面示例：不要这样做

**❌ 错误——这些在测试实现，而不是需求：**

```markdown
- [ ] CHK001 - 验证落地页显示 3 张剧集卡片 [Spec §FR-001]
- [ ] CHK002 - 测试桌面端悬停状态是否正常工作 [Spec §FR-003]
- [ ] CHK003 - 确认点击 logo 能跳转到首页 [Spec §FR-010]
- [ ] CHK004 - 检查相关剧集区块显示 3-5 个条目 [Spec §FR-005]
```

**✅ 正确——这些在测试需求质量：**


## 执行后检查

**检查扩展钩子（生成检查清单之后）**：
检查项目根目录是否存在 `.specify/extensions.yml`。
- 如果存在，读取它并查找 `hooks.after_checklist` 键下的条目
- 如果 YAML 无法解析或格式无效，静默跳过钩子检查并正常继续
- 过滤掉 `enabled` 显式为 `false` 的钩子。没有 `enabled` 字段的钩子默认视为启用。
- 对其余每个钩子，**不要**尝试解释或求值钩子的 `condition` 表达式：
  - 如果钩子没有 `condition` 字段，或该字段为 null/空，将钩子视为可执行
  - 如果钩子定义了非空的 `condition`，跳过该钩子，把条件求值留给 HookExecutor 实现
- 对每个可执行的钩子，根据其 `optional` 标志输出以下内容：
  - **可选钩子**（`optional: true`）：
    ```
    ## 扩展钩子

    **可选钩子**：{extension}
    命令：`/{command}`
    描述：{description}

    提示词：{prompt}
    执行方式：`/{command}`
    ```
  - **强制钩子**（`optional: false`）：
    ```
    ## 扩展钩子

    **自动钩子**：{extension}
    正在执行：`/{command}`
    EXECUTE_COMMAND: {command}
    ```
    输出上面的代码块之后，你必须真正调用该钩子，并等待它完成后再继续。请像你在本智能体/会话中亲自运行该命令那样运行它（实际调用方式可能与上面所示的字面 `{command}` id 不同，例如技能模式的智能体会以 `/skill:speckit-...` 或 `$speckit-...` 的形式运行）。仅输出代码块本身并不会运行钩子。
- 如果没有注册任何钩子，或 `.specify/extensions.yml` 不存在，静默跳过
