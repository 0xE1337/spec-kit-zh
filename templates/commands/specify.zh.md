## 用户输入

```text
$ARGUMENTS
```

在继续之前，你**必须**考虑用户输入（如果不为空）。

## 执行前检查

**检查扩展钩子（编写规范之前）**：
- 检查项目根目录是否存在 `.specify/extensions.yml`。
- 如果存在，读取它并查找 `hooks.before_specify` 键下的条目
- 如果 YAML 无法解析或格式无效，静默跳过钩子检查并正常继续
- 过滤掉 `enabled` 显式为 `false` 的钩子。没有 `enabled` 字段的钩子默认视为启用。
- 对其余每个钩子，**不要**尝试解释或求值钩子的 `condition` 表达式：
  - 如果钩子没有 `condition` 字段，或该字段为 null/空，将钩子视为可执行
  - 如果钩子定义了非空的 `condition`，跳过该钩子，把条件求值留给 HookExecutor 实现
- 对每个可执行的钩子，根据其 `optional` 标志输出以下内容：
  - **可选钩子**（`optional: true`）：
    ```
    ## 扩展钩子

    **可选前置钩子**：{extension}
    命令：`/{command}`
    描述：{description}

    提示词：{prompt}
    执行方式：`/{command}`
    ```
  - **强制钩子**（`optional: false`）：
    ```
    ## 扩展钩子

    **自动前置钩子**：{extension}
    正在执行：`/{command}`
    EXECUTE_COMMAND: {command}

    等待钩子命令的结果后，再继续执行"概要"部分。
    ```
    输出上面的代码块之后，你必须真正调用该钩子，并等待它完成后再继续。请像你在本智能体/会话中亲自运行该命令那样运行它（实际调用方式可能与上面所示的字面 `{command}` id 不同，例如技能模式的智能体会以 `/skill:speckit-...` 或 `$speckit-...` 的形式运行）。仅输出代码块本身并不会运行钩子。
- 如果没有注册任何钩子，或 `.specify/extensions.yml` 不存在，静默跳过

## 概要

用户在触发消息中输入在 `__SPECKIT_COMMAND_SPECIFY__` 之后的文本**就是**功能描述。请假定你在本对话中始终能拿到它，即使下方出现字面的 `{ARGS}`。除非用户提供的命令为空，否则不要要求用户重复输入。

基于该功能描述，执行以下操作：

1. **为功能生成一个简洁的短名称**（2-4 个单词）：
   - 分析功能描述并提取最有意义的关键词
   - 创建一个能抓住功能本质的 2-4 个单词的短名称
   - 尽量使用"动作-名词"格式（例如 "add-user-auth"、"fix-payment-bug"）
   - 保留技术术语和缩写（OAuth2、API、JWT 等）
   - 保持简洁，但要有足够的描述性，让人一眼就能看懂这个功能
   - 示例：
     - "我想添加用户认证" → "user-auth"
     - "为 API 实现 OAuth2 集成" → "oauth2-api-integration"
     - "创建一个分析仪表盘" → "analytics-dashboard"
     - "修复支付处理超时的 bug" → "fix-payment-timeout"

2. **创建分支**（可选，通过钩子）：

   如果上面"执行前检查"中的 `before_specify` 钩子已成功运行，它会创建/切换到一个 git 分支，并输出包含 `BRANCH_NAME` 和 `FEATURE_NUM` 的 JSON。记下这些值以备参考，但分支名并**不**决定规范目录名。

   如果用户显式提供了 `GIT_BRANCH_NAME`，把它透传给钩子，让分支脚本把这个值原样用作分支名（绕过所有前缀/后缀生成逻辑）。

3. **创建规范功能目录**：

   除非用户显式提供 `SPECIFY_FEATURE_DIRECTORY`，否则规范默认存放在 `specs/` 目录下。

   **`SPECIFY_FEATURE_DIRECTORY` 的解析顺序**：
   1. 如果用户显式提供了 `SPECIFY_FEATURE_DIRECTORY`（例如通过环境变量、参数或配置），原样使用
   2. 否则，在 `specs/` 下自动生成：
      - 检查 `.specify/init-options.json` 中的 `feature_numbering`（首选）或 `branch_numbering`（已弃用，仅用于迁移——将在未来版本中移除）
      - 如果是 `"timestamp"`：前缀为 `YYYYMMDD-HHMMSS`（当前时间戳）
      - 如果是 `"sequential"` 或该配置不存在：前缀为 `NNN`（扫描 `specs/` 中已有目录后的下一个可用 3 位数字）
      - 构造目录名：`<prefix>-<short-name>`（例如 `003-user-auth` 或 `20260319-143022-user-auth`）
      - 把 `SPECIFY_FEATURE_DIRECTORY` 设为 `specs/<directory-name>`
      - 如果使用的是 `branch_numbering`（且 `feature_numbering` 不存在），输出一行警告："⚠️ init-options.json 中的 `branch_numbering` 已弃用。请重命名为 `feature_numbering`。"

   **创建目录和规范文件**：
   - `mkdir -p SPECIFY_FEATURE_DIRECTORY`
   - 通过 Spec Kit 的预设/模板解析栈解析当前生效的 `spec-template`（等价于 `specify preset resolve spec-template`）
   - 把解析出的 `spec-template` 文件复制到 `SPECIFY_FEATURE_DIRECTORY/spec.md` 作为起点
   - 把 `SPEC_FILE` 设为 `SPECIFY_FEATURE_DIRECTORY/spec.md`
   - 把解析出的路径持久化到 `.specify/feature.json`：
     ```json
     {
       "feature_directory": "<解析出的功能目录>"
     }
     ```
     写入实际解析出的目录路径值（例如 `specs/003-user-auth`），而不是字面字符串 `SPECIFY_FEATURE_DIRECTORY`。
     这让下游命令（`__SPECKIT_COMMAND_PLAN__`、`__SPECKIT_COMMAND_TASKS__` 等）无需依赖 git 分支命名约定就能定位功能目录。

   **重要**：
   - 每次 `__SPECKIT_COMMAND_SPECIFY__` 调用只能创建一个功能
   - 规范目录名和 git 分支名相互独立——它们可以相同，但这由用户决定
   - 规范目录和文件始终由本命令创建，绝不由钩子创建

4. 加载解析出的当前生效 `spec-template` 文件，了解必需的章节。

5. **如果存在**：加载 `/memory/constitution.md`，获取项目原则和治理约束。

6. 遵循以下执行流程：
    1. 从参数中解析用户描述
       如果为空：ERROR "未提供功能描述"
    2. 从描述中提取关键概念
       识别：参与者（actors）、动作、数据、约束
    3. 对不明确的方面：
       - 基于上下文和行业标准做出合理推断
       - 仅在满足以下条件时才标记 [NEEDS CLARIFICATION: 具体问题]：
         - 该选择会显著影响功能范围或用户体验
         - 存在多种合理解释，且各自影响不同
         - 不存在合理的默认值
       - **限制：[NEEDS CLARIFICATION] 标记总数最多 3 个**
       - 按影响给澄清项排优先级：范围 > 安全/隐私 > 用户体验 > 技术细节
    4. 填写"用户场景与测试"（User Scenarios & Testing）章节
       如果没有清晰的用户流程：ERROR "无法确定用户场景"
    5. 生成功能需求（Functional Requirements）
       每条需求都必须可测试
       对未指明的细节使用合理默认值（把假设记录在"假设"（Assumptions）章节）
    6. 定义成功标准（Success Criteria）
       给出可度量、技术无关的结果
       同时包含定量指标（时间、性能、数量）和定性度量（用户满意度、任务完成情况）
       每条标准都必须在不了解实现细节的情况下可验证
    7. 识别关键实体（Key Entities）（如果涉及数据）
    8. 返回：SUCCESS（规范就绪，可进入规划）

7. 使用模板结构把规范写入 SPEC_FILE，用从功能描述（参数）中得出的具体细节替换占位符，同时保持章节顺序和标题不变。

8. **规范质量验证**：写完初始规范后，对照质量标准进行验证：

   a. **创建规范质量检查清单**：使用检查清单模板结构，在 `SPECIFY_FEATURE_DIRECTORY/checklists/requirements.md` 生成一个检查清单文件，包含以下验证条目：

      ```markdown
      # 规范质量检查清单：[FEATURE NAME]

      **目的**：在进入规划之前验证规范的完整性和质量
      **创建时间**：[DATE]
      **功能**：[Link to spec.md]

      ## 内容质量

      - [ ] 无实现细节（语言、框架、API）
      - [ ] 聚焦用户价值和业务需求
      - [ ] 面向非技术相关方撰写
      - [ ] 所有必填章节已完成

      ## 需求完整性

      - [ ] 没有遗留 [NEEDS CLARIFICATION] 标记
      - [ ] 需求可测试且无歧义
      - [ ] 成功标准可度量
      - [ ] 成功标准与技术无关（不含实现细节）
      - [ ] 所有验收场景已定义
      - [ ] 边界情况已识别
      - [ ] 范围界定清晰
      - [ ] 依赖与假设已识别

      ## 功能就绪度

      - [ ] 所有功能需求都有清晰的验收标准
      - [ ] 用户场景覆盖主要流程
      - [ ] 功能满足"成功标准"中定义的可度量结果
      - [ ] 没有实现细节泄漏进规范

      ## 备注

      - 标记为未完成的条目需要先更新规范，才能进入 `__SPECKIT_COMMAND_CLARIFY__` 或 `__SPECKIT_COMMAND_PLAN__`
      ```

   b. **运行验证检查**：对照每个检查清单条目审查规范：
      - 对每个条目，判定其通过还是失败
      - 记录发现的具体问题（引用相关的规范章节）

   c. **处理验证结果**：

      - **如果所有条目都通过**：把检查清单标记为完成，并进入"强制性执行后钩子"部分

      - **如果有条目失败（不含 [NEEDS CLARIFICATION]）**：
        1. 列出失败的条目和具体问题
        2. 更新规范以解决每个问题
        3. 重新运行验证，直到所有条目通过（最多迭代 3 次）
        4. 如果迭代 3 次后仍有失败，把剩余问题记录在检查清单备注中并警告用户

      - **如果仍有 [NEEDS CLARIFICATION] 标记**：
        1. 从规范中提取所有 [NEEDS CLARIFICATION: ...] 标记
        2. **数量检查**：如果标记超过 3 个，只保留最关键的 3 个（按范围/安全/用户体验影响排序），其余做出合理推断
        3. 对每个需要澄清的问题（最多 3 个），按以下格式向用户呈现选项：

           ```markdown
           ## 问题 [N]：[主题]

           **上下文**：[引用相关的规范章节]

           **我们需要了解**：[NEEDS CLARIFICATION 标记中的具体问题]

           **建议答案**：

           | 选项 | 答案 | 影响 |
           |--------|--------|--------------|
           | A      | [第一个建议答案] | [对功能意味着什么] |
           | B      | [第二个建议答案] | [对功能意味着什么] |
           | C      | [第三个建议答案] | [对功能意味着什么] |
           | 自定义 | 提供你自己的答案 | [说明如何提供自定义输入] |

           **你的选择**：_[等待用户回复]_
           ```

        4. **关键——表格格式**：确保 markdown 表格格式正确：
           - 使用一致的间距，管道符对齐
           - 每个单元格内容两侧都要有空格：`| Content |` 而不是 `|Content|`
           - 表头分隔行至少要有 3 个连字符：`|--------|`
           - 确认表格在 markdown 预览中能正确渲染
        5. 按顺序给问题编号（Q1、Q2、Q3——总共最多 3 个）
        6. 一次性呈现所有问题，然后再等待回复
        7. 等待用户对所有问题给出选择（例如 "Q1: A, Q2: 自定义 - [具体内容], Q3: B"）
        8. 用用户选择或提供的答案替换每个 [NEEDS CLARIFICATION] 标记，更新规范
        9. 所有澄清项都解决后，重新运行验证

   d. **更新检查清单**：每轮验证迭代后，用当前的通过/失败状态更新检查清单文件

## 强制性执行后钩子

- 如果不存在，或 `hooks.after_specify` 下没有注册任何钩子，直接跳到"完成报告"。
- 如果存在，读取它并查找 `hooks.after_specify` 键下的条目。
- 如果 YAML 无法解析或格式无效，静默跳过钩子检查并继续执行"完成报告"。
- 过滤掉 `enabled` 显式为 `false` 的钩子。没有 `enabled` 字段的钩子默认视为启用。

  - 如果钩子没有 `condition` 字段，或该字段为 null/空，将钩子视为可执行
  - 如果钩子定义了非空的 `condition`，跳过该钩子，把条件求值留给 HookExecutor 实现
- 对每个可执行的钩子，根据其 `optional` 标志输出以下内容：
  - **强制钩子**（`optional: false`）——**你必须为每个强制钩子输出 `EXECUTE_COMMAND:`**：

## 完成报告

向用户报告完成情况，包含：
- `SPECIFY_FEATURE_DIRECTORY`——功能目录路径
- `SPEC_FILE`——规范文件路径
- 检查清单结果摘要
- 下一阶段（`__SPECKIT_COMMAND_CLARIFY__` 或 `__SPECKIT_COMMAND_PLAN__`）的就绪情况

**注意**：分支创建由 `before_specify` 钩子（git 扩展）处理。规范目录和文件的创建始终由本核心命令处理。

## 快速指南

- 聚焦用户需要**什么**以及**为什么**。
- 避免"怎么实现"（不谈技术栈、API、代码结构）。
- 面向业务相关方撰写，而不是开发者。
- 不要创建任何内嵌在规范里的检查清单。那将由单独的命令处理。

### 章节要求

- **必填章节**：每个功能都必须完成
- **可选章节**：仅在与该功能相关时才包含
- 章节不适用时，直接整节删除（不要留成 "N/A"）

### 面向 AI 生成

根据用户提示词创建这份规范时：

1. **做出合理推断**：利用上下文、行业标准和常见模式填补空白
2. **记录假设**：把采用的合理默认值记录在"假设"（Assumptions）章节
3. **限制澄清数量**：[NEEDS CLARIFICATION] 标记最多 3 个——只用于满足以下条件的关键决策：
   - 显著影响功能范围或用户体验
   - 存在多种合理解释且各自影响不同
   - 缺乏任何合理默认值
4. **澄清优先级**：范围 > 安全/隐私 > 用户体验 > 技术细节
5. **像测试人员一样思考**：任何模糊的需求都应该无法通过"可测试且无歧义"这条检查项
6. **常见的需要澄清的领域**（仅当不存在合理默认值时）：
   - 功能范围与边界（包含/排除特定用例）
   - 用户类型与权限（如果可能存在多种相互冲突的解释）
   - 安全/合规需求（当涉及重大法律/财务影响时）

**合理默认值的示例**（不要就这些提问）：

- 数据保留：该领域的行业标准做法
- 性能目标：标准的 Web/移动应用预期，除非另有说明
- 错误处理：用户友好的消息，配合适当的回退机制
- 认证方式：Web 应用采用标准的基于会话的认证或 OAuth2
- 集成模式：使用与项目相称的模式（Web 服务用 REST/GraphQL，库用函数调用，工具用 CLI 参数等）

### 成功标准指南

成功标准必须：

1. **可度量**：包含具体指标（时间、百分比、数量、比率）
2. **技术无关**：不提及框架、语言、数据库或工具
3. **以用户为中心**：从用户/业务视角描述结果，而不是系统内部
4. **可验证**：无需了解实现细节即可测试/验证

**好的示例**：

- "用户能在 3 分钟内完成结账"
- "系统支持 10,000 并发用户"
- "95% 的搜索在 1 秒内返回结果"
- "任务完成率提升 40%"

**差的示例**（聚焦实现）：

- "API 响应时间低于 200ms"（太技术化，应使用"用户能立即看到结果"）
- "数据库能处理 1000 TPS"（实现细节，应使用面向用户的指标）
- "React 组件渲染高效"（绑定特定框架）
- "Redis 缓存命中率高于 80%"（绑定特定技术）

## 完成标准

- [ ] 规范已写入 `SPEC_FILE`，并通过质量检查清单验证
- [ ] 扩展钩子已按上文"强制性执行后钩子"的规则分发或跳过
- [ ] 已向用户报告完成情况，包含功能目录、规范文件路径和检查清单结果
