## 执行前检查

**检查扩展钩子（澄清之前）**：
- 检查项目根目录是否存在 `.specify/extensions.yml`。
- 如果存在，读取它并查找 `hooks.before_clarify` 键下的条目
- 如果 YAML 无法解析或格式无效，静默跳过钩子检查并正常继续
- 过滤掉 `enabled` 显式为 `false` 的钩子。没有 `enabled` 字段的钩子默认视为启用。
- 对其余每个钩子，**不要**尝试解释或求值钩子的 `condition` 表达式：
  - 如果钩子没有 `condition` 字段，或该字段为 null/空，将钩子视为可执行
  - 如果钩子定义了非空的 `condition`，跳过该钩子，把条件求值留给 HookExecutor 实现
- 对每个可执行的钩子，根据其 `optional` 标志输出以下内容：
  - **可选钩子**（`optional: true`）：
    ```
    ## 扩展钩子

    **可选前置钩子**：{extension}
    命令：`/{command}`
    描述：{description}

    提示词：{prompt}
    执行方式：`/{command}`
    ```
  - **强制钩子**（`optional: false`）：
    ```
    ## 扩展钩子

    **自动前置钩子**：{extension}
    正在执行：`/{command}`
    EXECUTE_COMMAND: {command}

    等待钩子命令的结果后，再继续执行"概要"部分。
    ```
    输出上面的代码块之后，你必须真正调用该钩子，并等待它完成后再继续。请像你在本智能体/会话中亲自运行该命令那样运行它（实际调用方式可能与上面所示的字面 `{command}` id 不同，例如技能模式的智能体会以 `/skill:speckit-...` 或 `$speckit-...` 的形式运行）。仅输出代码块本身并不会运行钩子。
- 如果没有注册任何钩子，或 `.specify/extensions.yml` 不存在，静默跳过

## 概要

目标：检测并减少当前功能规范中的歧义或缺失的决策点，并把澄清结果直接记录到规范文件中。

注意：本澄清工作流应当在调用 `__SPECKIT_COMMAND_PLAN__` **之前**运行（并完成）。如果用户明确表示要跳过澄清（例如探索性 spike），你可以继续，但必须警告下游返工风险会增加。

执行步骤：

1. 在仓库根目录运行**一次** `{SCRIPT}`（组合使用 `--json --paths-only` 模式 / `-Json -PathsOnly`）。解析最小 JSON 载荷中的字段：
   - `FEATURE_DIR`
   - `FEATURE_SPEC`
   - （可选：捕获 `IMPL_PLAN`、`TASKS`，供未来的链式流程使用。）
   - 如果 JSON 解析失败，中止并指示用户重新运行 `__SPECKIT_COMMAND_SPECIFY__`，或检查功能分支环境是否正常。
   - 参数中含单引号（如 "I'm Groot"）时，使用转义语法：例如 'I'\''m Groot'（或尽可能用双引号："I'm Groot"）。

2. **如果存在**：加载 `/memory/constitution.md`，获取项目原则和治理约束。

3. 加载当前规范文件。使用以下分类体系执行结构化的歧义与覆盖度扫描。对每个类别标记状态：清晰（Clear）/部分（Partial）/缺失（Missing）。生成一份用于优先级排序的内部覆盖度映射（除非最终一个问题都不会问，否则不要输出原始映射）。

   功能范围与行为（Functional Scope & Behavior）：
   - 核心用户目标与成功标准
   - 明确的范围外（out-of-scope）声明
   - 用户角色/画像（personas）的区分

   领域与数据模型（Domain & Data Model）：
   - 实体、属性、关系
   - 标识与唯一性规则
   - 生命周期/状态流转
   - 数据量/规模假设

   交互与用户体验流程（Interaction & UX Flow）：
   - 关键用户旅程/操作序列
   - 错误/空/加载状态
   - 无障碍或本地化说明

   非功能质量属性（Non-Functional Quality Attributes）：
   - 性能（延迟、吞吐量目标）
   - 可扩展性（水平/垂直、上限）
   - 可靠性与可用性（正常运行时间、恢复预期）
   - 可观测性（日志、指标、链路追踪信号）
   - 安全与隐私（认证/授权、数据保护、威胁假设）
   - 合规/监管约束（如有）

   集成与外部依赖（Integration & External Dependencies）：
   - 外部服务/API 及其失败模式
   - 数据导入/导出格式
   - 协议/版本假设

   边界情况与失败处理（Edge Cases & Failure Handling）：
   - 负面场景
   - 速率限制/限流
   - 冲突解决（例如并发编辑）

   约束与取舍（Constraints & Tradeoffs）：
   - 技术约束（语言、存储、托管）
   - 明确的取舍或被否决的备选方案

   术语与一致性（Terminology & Consistency）：
   - 规范的术语表用词
   - 应避免的同义词/已弃用术语

   完成信号（Completion Signals）：
   - 验收标准的可测试性
   - 可度量的"完成定义"（Definition of Done）式指标

   杂项/占位（Misc / Placeholders）：
   - TODO 标记/未决定项
   - 缺乏量化的模糊形容词（如"健壮"、"直观"）

   对状态为"部分"或"缺失"的每个类别，添加一个候选提问机会，除非：
   - 澄清不会实质性地改变实现或验证策略
   - 该信息更适合推迟到规划阶段（在内部记录即可）

4. （在内部）生成一个按优先级排序的候选澄清问题队列（最多 5 个）。不要一次性全部输出。应用以下约束：
    - 整个会话总共最多 5 个问题。
    - 每个问题必须满足以下二者之一：
       - 可以用简短的多选题回答（2-5 个互斥的不同选项），或
       - 可以用一个词/短语回答（明确限制："5 个词以内作答"）。
    - 只纳入答案会实质影响架构、数据建模、任务分解、测试设计、用户体验行为、运维就绪或合规验证的问题。
    - 确保类别覆盖均衡：优先覆盖影响最大的未解决类别；当某个高影响领域（例如安全态势）尚未解决时，不要问两个低影响问题。
    - 排除已经回答过的问题、琐碎的风格偏好，以及计划层面的执行细节（除非它阻塞正确性）。
    - 优先选择能降低下游返工风险或防止验收测试偏差的澄清。
    - 如果未解决的类别超过 5 个，按（影响 × 不确定性）启发式选出前 5 个。

5. 顺序提问循环（交互式）：
    - 每次只呈现一个问题。
    - 多选题：
       - **分析所有选项**，并基于以下因素判断**最合适的选项**：
          - 该项目类型的最佳实践
          - 类似实现中的常见模式
          - 降低风险（安全、性能、可维护性）
          - 与规范中可见的项目目标或约束保持一致
       - 把你的**推荐选项醒目地放在最上方**，并给出清晰的理由（用 1-2 句话解释为什么这是最佳选择）。
       - 格式：`**推荐：** 选项 [X] - <理由>`
       - 然后把所有选项渲染为 Markdown 表格：

       | 选项 | 描述 |
       |--------|-------------|
       | A | <选项 A 描述> |
       | B | <选项 B 描述> |
       | C | <选项 C 描述>（按需增加 D/E，最多 5 个） |
       | 简答 | 提供另外的简短答案（5 个词以内）（仅在适合自由作答时包含此行） |

       - 表格之后补充一句：`你可以回复选项字母（例如 "A"），或回复 "yes"/"recommended" 接受推荐，也可以提供你自己的简短答案。`
    - 简答式问题（不存在有意义的离散选项）：
       - 基于最佳实践和上下文给出你的**建议答案**。
       - 格式：`**建议：** <你提议的答案> - <简要理由>`
       - 然后输出：`格式：简短回答（5 个词以内）。你可以回复 "yes" 或 "suggested" 接受建议，也可以提供你自己的答案。`
    - 用户回答之后：
       - 如果用户回复 "yes"、"recommended" 或 "suggested"，使用你先前给出的推荐/建议作为答案。
       - 否则，校验答案能对应到某个选项，或符合 5 个词以内的限制。
       - 如果仍有歧义，追问一次快速消歧（计数仍归属同一个问题；不推进到下一题）。
       - 答案满意后，把它记录到工作记忆中（先不要写入磁盘），并进入队列中的下一个问题。
    - 满足以下任一条件时停止提问：
       - 所有关键歧义提前解决（队列中剩余的问题变得不必要），或
       - 用户发出完成信号（"done"、"good"、"no more"），或
       - 已问满 5 个问题。
    - 绝不提前透露队列中后续的问题。
    - 如果一开始就没有有效的问题，立即报告没有关键歧义。

6. 每接受一个答案后立即整合（增量更新方式）：
    - 维护规范的内存表示（开始时加载一次），以及原始文件内容。
    - 本次会话第一次整合答案时：
       - 确保存在 `## Clarifications` 小节（如果缺失，按照规范模板，在最高层级的上下文/概览章节之后创建）。
       - 在其下创建（如果不存在）今天的 `### Session YYYY-MM-DD` 子标题。
    - 在答案被接受后立即追加一行：`- Q: <问题> → A: <最终答案>`。
    - 然后立即把该澄清应用到最合适的章节：
       - 功能性歧义 → 更新或新增"功能需求"（Functional Requirements）中的条目。
       - 用户交互/参与者区分 → 用澄清后的角色、约束或场景更新"用户故事"（User Stories）或"参与者"（Actors）小节（如果存在）。
       - 数据形态/实体 → 更新"数据模型"（Data Model）（添加字段、类型、关系），保持原有顺序；简洁注明新增的约束。
       - 非功能约束 → 在"成功标准 > 可度量结果"（Success Criteria > Measurable Outcomes）中添加/修改可度量标准（把模糊形容词转换为指标或明确目标）。
       - 边界情况/负面流程 → 在"边界情况/错误处理"（Edge Cases / Error Handling）下新增条目（如果模板为其提供了占位，则创建该小节）。
       - 术语冲突 → 在整个规范中统一为规范术语；确有必要时才保留原词，方式是加注一次 `（曾称"X"）`。
    - 如果该澄清使先前某条模糊表述失效，替换那条表述而不是重复添加；不留任何过时的矛盾文本。
    - 每次整合后立即保存规范文件（原子覆盖写入），把上下文丢失的风险降到最低。
    - 保持格式：不要重排无关章节；保持标题层级不变。
    - 每条插入的澄清保持精简、可测试（避免叙事漂移）。

7. 验证（每次写入后执行一次，最后再整体执行一遍）：
   - 澄清会话中每个已接受的答案恰好对应一条记录（无重复）。
   - 已问（已接受）的问题总数 ≤ 5。
   - 更新后的章节中不再残留新答案本应解决的模糊占位表述。
   - 不再存在相互矛盾的旧表述（扫描并移除如今已失效的备选说法）。
   - Markdown 结构有效；只允许新增这些标题：`## Clarifications`、`### Session YYYY-MM-DD`。
   - 术语一致性：所有更新过的章节使用同一个规范术语。

8. 把更新后的规范写回 `FEATURE_SPEC`。

9. **重新验证规范质量检查清单**（如果存在）：
   - 检查 `FEATURE_DIR/checklists/requirements.md` 是否存在。
   - 如果不存在，静默跳过此步骤。
   - 如果存在：
     1. 读取检查清单文件。
     2. 识别所有 GitHub 任务列表复选框行——即代码围栏之外匹配 `- [ ]`、`- [x]` 或 `- [X]` 的行（大小写不敏感，容忍嵌套条目的行首空白）。忽略所有其他内容（标题、备注、非复选框条目、元数据）。
     3. 对每个复选框行，把当前的标记状态（已勾选或未勾选）和条目文本记录到一份"变更前快照"列表中。
     4. 对照**更新后**的规范（即步骤 7 中刚保存的版本）重新评估每个复选框条目。
     5. 对每个复选框条目，只有当勾选状态确实发生变化时才更新：
        - 如果条目现在通过且原本未勾选：把 `[ ]` 改为 `[x]`。
        - 如果条目现在失败且原本已勾选：把 `[x]`/`[X]` 改为 `[ ]`。
        - 如果状态未变：保持标记原样（保留原有大小写，避免产生纯格式的 diff）。
     6. 保存更新后的检查清单文件。**只切换状态发生变化的复选框行的 `[ ]`/`[x]` 标记部分。**文件的所有其他内容——标题、元数据、备注、行顺序、空白——都必须保持不变，避免产生带噪音的 diff。
     7. 对比变更前快照与当前状态，为"完成报告"计算三个列表：
        - **新通过**：从未勾选变为已勾选的条目。
        - **回退**：从已勾选变为未勾选的条目。
        - **仍未勾选**：保持未勾选的条目。
     8. 以"已勾选/复选框总数"的形式记录变更前后的通过数（例如 "12/16 → 15/16 项通过"）。

行为规则：

- 如果没有发现有意义的歧义（或所有潜在问题都是低影响的），回复："未检测到值得正式澄清的关键歧义。"并建议继续推进。
- 如果规范文件缺失，指示用户先运行 `__SPECKIT_COMMAND_SPECIFY__`（不要在这里新建规范）。
- 已问问题总数绝不超过 5 个（针对单个问题的澄清重试不算新问题）。
- 避免猜测性的技术栈问题，除非缺少该信息会阻碍功能层面的清晰度。
- 尊重用户的提前终止信号（"stop"、"done"、"proceed"）。
- 如果因覆盖度已完整而未提问，输出一份紧凑的覆盖度摘要（所有类别均为"清晰"），然后建议进入下一步。
- 如果问题配额用尽时仍有未解决的高影响类别，在"已推迟"（Deferred）项下明确标出并说明理由。

## 完成报告

报告完成情况（在提问循环结束或提前终止之后）：
- 已提问并获答的问题数量。
- 更新后规范的路径。
- 触及的章节（列出名称）。
- 规范质量检查清单状态（如果重新验证了 `FEATURE_DIR/checklists/requirements.md`）：展示变更前后的通过数（例如 "规范质量检查清单：12/16 → 15/16 项通过"），并列出所有状态发生变化的条目——包括新勾选的（未勾选 → 已勾选）和所有回退的（已勾选 → 未勾选）。如果仍有未勾选的条目，把它们列为需要关注的领域。
- 覆盖度摘要表，列出分类体系中的每个类别及其状态：已解决（原为部分/缺失且已处理）、已推迟（超出问题配额或更适合规划阶段）、清晰（原本就已充分）、未解决（仍为部分/缺失但影响较低）。
- 如果仍有"未解决"或"已推迟"项，给出建议：是继续进入 `__SPECKIT_COMMAND_PLAN__`，还是稍后在计划完成之后再运行一次 `__SPECKIT_COMMAND_CLARIFY__`。
- 建议的下一个命令。
