# 任务清单：[FEATURE NAME]

**输入**：来自 `/specs/[###-feature-name]/` 的设计文档

**前置条件**：plan.md（必需）、spec.md（用户故事所必需）、research.md、data-model.md、contracts/

**测试**：下面的示例包含测试任务。测试是可选的——只有当功能规范明确要求时才包含。

**组织方式**：任务按用户故事分组，以便每个故事都能独立实现、独立测试。

## 格式：`[ID] [P?] [Story] 描述`

- **[P]**：可以并行运行（不同文件、无依赖）
- **[Story]**：该任务所属的用户故事（例如 US1、US2、US3）
- 描述中要包含确切的文件路径

## 路径约定

- **单一项目**：`src/`、`tests/` 位于仓库根目录
- **Web 应用**：`backend/src/`、`frontend/src/`
- **移动端**：`api/src/`、`ios/src/` 或 `android/src/`
- 下面展示的路径以单一项目为前提——请根据 plan.md 的结构调整

## 依赖与执行顺序

### 阶段依赖

- **准备（阶段 1）**：无依赖——可立即开始
- **基础（阶段 2）**：依赖准备阶段完成——阻塞所有用户故事
- **用户故事（阶段 3+）**：全部依赖基础阶段完成
  - 之后用户故事可以并行推进（如果人手允许）
  - 或按优先级顺序依次进行（P1 → P2 → P3）
- **打磨（最后阶段）**：依赖所有目标用户故事完成

### 用户故事依赖

- **用户故事 1（P1）**：基础（阶段 2）之后即可开始——不依赖其他故事
- **用户故事 2（P2）**：基础（阶段 2）之后即可开始——可能与 US1 集成，但应保持可独立测试
- **用户故事 3（P3）**：基础（阶段 2）之后即可开始——可能与 US1/US2 集成，但应保持可独立测试

### 每个用户故事内部

- 测试（如包含）必须先写好，并在实现之前处于失败状态
- 先模型，再服务
- 先服务，再端点
- 先核心实现，再集成
- 完成当前故事后再进入下一个优先级

### 并行机会

- 所有标注 [P] 的准备阶段任务可以并行运行
- 所有标注 [P] 的基础任务可以并行运行（在阶段 2 内）
- 基础阶段完成后，所有用户故事都可以并行启动（如果团队人力允许）
- 一个用户故事中所有标注 [P] 的测试可以并行运行
- 一个故事中标注 [P] 的模型可以并行运行
- 不同的用户故事可以由不同的团队成员并行推进

## 实现策略

### MVP 优先（只做用户故事 1）

1. 完成阶段 1：准备
2. 完成阶段 2：基础（关键——阻塞所有故事）
3. 完成阶段 3：用户故事 1
4. **停下来验证**：独立测试用户故事 1
5. 就绪后部署/演示

### 增量交付

1. 完成准备 + 基础 → 基础就绪
2. 加入用户故事 1 → 独立测试 → 部署/演示（MVP！）
3. 加入用户故事 2 → 独立测试 → 部署/演示
4. 加入用户故事 3 → 独立测试 → 部署/演示
5. 每个故事都在不破坏之前故事的前提下增加价值

### 团队并行策略

多名开发者协作时：

1. 团队一起完成准备 + 基础
2. 基础完成后：
   - 开发者 A：用户故事 1
   - 开发者 B：用户故事 2
   - 开发者 C：用户故事 3
3. 各故事独立完成、独立集成

## 说明

- [P] 任务 = 不同文件、无依赖
- [Story] 标签把任务映射到具体的用户故事，便于追溯
- 每个用户故事都应该能独立完成、独立测试
- 实现之前先确认测试是失败的
- 每完成一个任务或一组相关任务就提交一次
- 可以在任何检查点停下来，独立验证当前故事
- 避免：任务描述含糊、多个任务修改同一文件产生冲突、破坏独立性的跨故事依赖
