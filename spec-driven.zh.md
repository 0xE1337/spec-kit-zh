# 规范驱动开发（SDD）

## 权力反转

几十年来，代码一直是主角。规范服务于代码——它们是我们搭起来、等"真正的编码工作"开始后就丢弃的脚手架。我们写 PRD 来指导开发，写设计文档来支撑实现，画架构图来呈现系统。但这些始终从属于代码本身。代码就是真理，其他一切充其量只是良好的意愿。代码是事实来源，而代码一路向前，规范却很少能跟上。既然资产（代码）与实现本是一体，不从代码出发，就很难再造一个并行实现。

规范驱动开发（SDD）反转了这个权力结构。规范不再服务于代码——代码服务于规范。产品需求文档（PRD）不是实现的参考指南，而是生成实现的源头。技术计划不是给编码提供参考的文档，而是能产出代码的精确定义。这不是对软件构建方式的渐进式改良，而是对"什么在驱动开发"的根本性重新思考。

规范与实现之间的鸿沟，从软件开发诞生之日起就困扰着这个行业。我们试过用更好的文档、更详细的需求、更严格的流程来弥合它。这些方法之所以失败，是因为它们把鸿沟当作不可避免的事实来接受：只求收窄，从不指望消除。SDD 通过让规范以及由规范派生的具体实现计划变得可执行，消除了这道鸿沟。当规范和实现计划能生成代码时，就不存在鸿沟——只剩下转换。

这种转换如今之所以可行，是因为 AI 已经能理解并实现复杂的规范，还能创建详细的实现计划。但没有结构约束的原始 AI 生成只会产出混乱。SDD 通过规范和后续的实现计划提供了这种结构：它们足够精确、完整、无歧义，足以生成可运行的系统。规范成为主要产物，代码成为它在特定语言和框架中的表达（作为由实现计划推导出的实现）。

在这个新世界里，维护软件意味着演进规范。开发团队的意图用自然语言（"**意图驱动开发**"）、设计资产、核心原则和其他准则来表达。开发的**通用语（lingua franca）**上移到了更高的层次，代码则成为"最后一公里"的手段。

调试意味着修正生成了错误代码的规范及其实现计划。重构意味着为了清晰而重新组织结构。整个开发工作流围绕规范这一核心事实来源重新组织，实现计划和代码则是持续再生成的输出。给应用增加新功能，或者出于创造的天性再造一个并行实现，都意味着回到规范并创建新的实现计划。因此这个过程是 0 -> 1，(1', ..)，2，3，N。

开发团队得以专注于他们的创造力、实验和批判性思维。

## SDD 工作流实践

工作流始于一个想法——往往模糊而不完整。通过与 AI 的迭代对话，这个想法逐渐变成一份全面的 PRD。AI 会提出澄清问题、识别边界情况，并帮助定义精确的验收标准。传统开发中可能需要数天会议和文档工作的事情，如今在几小时专注的规范打磨中就能完成。这改变了传统的软件开发生命周期（SDLC）——需求和设计成为持续的活动，而不是离散的阶段。这也支撑了一种**团队流程**：经团队评审的规范被表达出来并纳入版本管理，在分支中创建，再被合并。

当产品经理更新验收标准时，实现计划会自动标记出受影响的技术决策。当架构师发现更好的模式时，PRD 会更新以反映新的可能性。

在整个规范制定过程中，研究智能体负责收集关键上下文。它们调查库的兼容性、性能基准和安全影响。组织级约束会被自动发现并应用——你们公司的数据库标准、认证要求和部署策略会无缝融入每一份规范。

AI 从 PRD 生成实现计划，把需求映射为技术决策。每一个技术选型都有记录在案的理由，每一个架构决策都能追溯到具体需求。在整个过程中，一致性校验持续提升质量。AI 会分析规范中的歧义、矛盾和缺口——不是一次性的关卡，而是持续进行的精炼。

一旦规范及其实现计划足够稳定，代码生成就可以开始，但它们不必"完整"。早期的生成可能是探索性的——检验这份规范在实践中是否说得通。领域概念变成数据模型，用户故事变成 API 端点，验收场景变成测试。这通过规范把开发和测试融合在一起——测试场景不是写完代码之后才补的，而是规范的一部分，由它同时生成实现和测试。

反馈循环不止于初始开发。生产环境的指标和事故不只是触发热修复——它们会更新规范，供下一次再生成使用。性能瓶颈变成新的非功能性需求，安全漏洞变成约束所有后续生成的限制条件。规范、实现与运行现实之间的这种迭代往复，正是真正的理解得以浮现的地方，也是传统 SDLC 转变为持续演进的地方。

## 为什么 SDD 正当其时

三个趋势让 SDD 不仅成为可能，而且成为必需：

第一，AI 能力已经跨过一个门槛：自然语言规范可以可靠地生成可运行的代码。这不是要取代开发者，而是通过自动化"从规范到实现"的机械翻译来放大他们的效能。它可以放大探索与创造，轻松支持"推倒重来"，也支持加法、减法和批判性思考。

第二，软件复杂度持续呈指数级增长。现代系统要集成数十个服务、框架和依赖。靠手工流程让所有这些部分与最初的意图保持一致，正变得越来越难。SDD 通过规范驱动的生成提供系统化的对齐。框架可能会演进为提供 AI 优先（而非人类优先）的支持，或者围绕可复用组件进行架构设计。

第三，变化的节奏在加速。如今需求变化的速度远超以往。转向（pivot）不再是例外——而是常态。现代产品开发要求基于用户反馈、市场状况和竞争压力快速迭代。传统开发把这些变化当作干扰：每次转向都要手工把变更传导到文档、设计和代码里。结果要么是缓慢谨慎、拖累速度的更新，要么是快速鲁莽、积累技术债的改动。

SDD 可以支持"假设推演"（what-if）/模拟类实验："如果为了推动多卖 T 恤的业务需求，我们要重新实现或改造这个应用，我们会如何实现并进行实验？"

SDD 把需求变更从障碍变成正常工作流。当规范驱动实现时，转向变成系统化的再生成，而不是手工重写。修改 PRD 里的一条核心需求，受影响的实现计划会自动更新；调整一个用户故事，对应的 API 端点会重新生成。这不只关乎初始开发——而是在不可避免的变化中保持工程速度。

## 核心原则

**规范作为通用语**：规范成为主要产物，代码成为它在特定语言和框架中的表达。维护软件意味着演进规范。

**可执行的规范**：规范必须足够精确、完整、无歧义，足以生成可运行的系统。这消除了意图与实现之间的鸿沟。

**持续精炼**：一致性校验持续进行，而不是一次性的关卡。AI 把分析规范中的歧义、矛盾和缺口当作一个持续的过程。

**研究驱动的上下文**：研究智能体在整个规范制定过程中收集关键上下文，调查技术选项、性能影响和组织级约束。

**双向反馈**：生产现实为规范演进提供输入。指标、事故和运维经验成为规范精炼的素材。

**分支式探索**：从同一份规范生成多种实现方案，探索不同的优化目标——性能、可维护性、用户体验、成本。

## 落地方式

今天实践 SDD，需要组合现有工具，并在整个过程中保持纪律。这套方法论可以借助以下手段实践：

- 用 AI 助手进行迭代式的规范打磨
- 用研究智能体收集技术上下文
- 用代码生成工具把规范转换为实现
- 用适配了规范优先工作流的版本控制系统
- 通过 AI 分析规范文档来做一致性检查

关键在于把规范当作事实来源，代码则是服务于规范的生成输出，而不是反过来。

## 用命令简化 SDD 流程

三个强大的命令显著增强了 SDD 方法论，它们把"规范 → 规划 → 任务"的工作流自动化了：

### `/speckit.specify` 命令

这个命令把一句简单的功能描述（用户提示词）转换成一份完整、结构化的规范，并自动完成仓库管理：

1. **自动功能编号**：扫描现有规范，确定下一个功能编号（例如 001、002、003……直到 1000——超过 3 位数会自动扩展）
2. **分支创建**：根据你的描述生成语义化的分支名并自动创建分支
3. **基于模板的生成**：复制功能规范模板，并按你的需求进行定制
4. **目录结构**：为所有相关文档创建规范的 `specs/[branch-name]/` 结构

### `/speckit.plan` 命令

功能规范就绪后，这个命令会创建一份全面的实现计划：

1. **规范分析**：阅读并理解功能需求、用户故事和验收标准
2. **宪章合规**：确保与项目宪章和架构原则保持一致
3. **技术转换**：把业务需求转换为技术架构和实现细节
4. **详细文档**：为数据模型、API 契约和测试场景生成配套文档
5. **快速上手验证**：产出一份快速上手指南，记录关键的验证场景

### `/speckit.tasks` 命令

计划创建后，这个命令会分析计划及相关设计文档，生成可执行的任务列表：

1. **输入**：读取 `plan.md`（必需），以及存在时的 `data-model.md`、`contracts/` 和 `research.md`
2. **任务推导**：把契约、实体和场景转换为具体任务
3. **并行化**：把相互独立的任务标记为 `[P]`，并列出安全的并行分组
4. **输出**：在功能目录下写入 `tasks.md`，可直接交给 Task 智能体执行

### 示例：构建一个聊天功能

以下展示这些命令如何改变传统开发工作流：

**传统方式：**

```text
1. 在文档里写一份 PRD（2-3 小时）
2. 编写设计文档（2-3 小时）
3. 手工搭建项目结构（30 分钟）
4. 编写技术规范（3-4 小时）
5. 制定测试计划（2 小时）
合计：约 12 小时的文档工作
```

**SDD 命令方式：**

```bash
# 第 1 步：创建功能规范（5 分钟）
/speckit.specify 实时聊天系统，支持消息历史和用户在线状态

# 这会自动：
# - 创建分支 "003-chat-system"
# - 生成 specs/003-chat-system/spec.md
# - 填入结构化的需求

# 第 2 步：生成实现计划（5 分钟）
/speckit.plan 用 WebSocket 做实时消息，PostgreSQL 存消息历史，Redis 维护在线状态

# 第 3 步：生成可执行任务（5 分钟）
/speckit.tasks

# 这会自动创建：
# - specs/003-chat-system/plan.md
# - specs/003-chat-system/research.md（WebSocket 库对比）
# - specs/003-chat-system/data-model.md（Message 和 User 的模式）
# - specs/003-chat-system/contracts/（WebSocket 事件、REST 端点）
# - specs/003-chat-system/quickstart.md（关键验证场景）
# - specs/003-chat-system/tasks.md（从计划推导出的任务列表）
```

15 分钟内，你就拥有了：

- 一份带用户故事和验收标准的完整功能规范
- 一份包含技术选型及其理由的详细实现计划
- 可直接用于代码生成的 API 契约和数据模型
- 覆盖自动化与手工测试的全面测试场景
- 全部文档都在功能分支中妥善纳入版本管理

### 结构化自动化的力量

这些命令不只是节省时间——它们强制保证一致性和完整性：

1. **不遗漏细节**：模板确保每个方面都被考虑到，从非功能性需求到错误处理
2. **决策可追溯**：每个技术选择都能链接回具体需求
3. **活文档**：规范与代码保持同步，因为代码就是由规范生成的
4. **快速迭代**：修改需求并重新生成计划只需几分钟，而不是几天

这些命令把规范当作可执行的产物而非静态文档，从而践行了 SDD 原则。它们把规范流程从"不得不做的苦差事"变成了驱动开发的核心动力。

### 模板驱动的质量：结构如何约束 LLM 产出更好的结果

这些命令真正的威力不只在自动化，更在于模板如何引导 LLM 的行为，产出更高质量的规范。模板扮演着精心设计的提示词的角色，以富有成效的方式约束 LLM 的输出：

#### 1. **防止过早引入实现细节**

功能规范模板明确指示：

```text
- ✅ 聚焦于用户需要什么（WHAT）以及为什么需要（WHY）
- ❌ 避免写怎么实现（HOW）——不写技术栈、API、代码结构
```

这个约束迫使 LLM 保持恰当的抽象层次。当 LLM 可能下意识地跳到"用 React 加 Redux 实现"时，模板会让它继续聚焦在"用户需要数据的实时更新"上。这种分离确保即便实现技术更迭，规范依然稳定。

#### 2. **强制显式标注不确定性**

两个模板都强制使用 `[NEEDS CLARIFICATION]` 标记：

```text
从用户提示词创建这份规范时：
1. **标记所有歧义**：使用 [NEEDS CLARIFICATION: 具体问题]
2. **不要猜**：如果提示词没有说明某件事，就把它标记出来
```

这防止了 LLM 的常见行为：做出貌似合理但可能错误的假设。LLM 不能猜测"登录系统"用的是邮箱/密码认证，而必须把它标记为 `[NEEDS CLARIFICATION: 未指定认证方式——邮箱/密码、SSO 还是 OAuth？]`。

#### 3. **通过检查清单进行结构化思考**

模板包含全面的检查清单，充当规范的"单元测试"：

```markdown
### 需求完整性

- [ ] 不再残留 [NEEDS CLARIFICATION] 标记
- [ ] 需求可测试且无歧义
- [ ] 成功标准可度量
```

这些检查清单迫使 LLM 系统性地自查输出，抓住那些原本可能溜走的缺口。这相当于给了 LLM 一套质量保证框架。

#### 4. **通过关卡执行宪章合规**

实现计划模板通过阶段关卡强制执行架构原则：

```markdown
### 阶段 -1：实现前关卡

#### 简单性关卡（第七条）

- [ ] 项目数量 ≤3 个？
- [ ] 没有为未来过度设计？

#### 反抽象关卡（第八条）

- [ ] 直接使用框架能力？
- [ ] 单一模型表示？
```

这些关卡通过让 LLM 显式论证每一分复杂度来防止过度设计。关卡不通过时，LLM 必须在"复杂度跟踪"（Complexity Tracking）小节记录原因，为架构决策建立问责。

#### 5. **分层的细节管理**

模板强制执行合理的信息架构：

```text
**重要**：这份实现计划应保持高层视角、易于阅读。
任何代码示例、详细算法或大段的技术规范内容，
都必须放到对应的 `implementation-details/` 文件中
```

这避免了规范沦为不可读的代码堆砌这一常见问题。LLM 学会维持恰当的细节层级，把复杂内容抽取到独立文件，让主文档保持易于导航。

#### 6. **测试先行的思维**

实现模板强制测试先行的开发方式：

```text
### 文件创建顺序
1. 创建 `contracts/`，写入 API 规范
2. 按顺序创建测试文件：契约测试 → 集成测试 → e2e 测试 → 单元测试
3. 创建源码文件，让测试通过
```

这个顺序约束确保 LLM 在实现之前先思考可测试性和契约，从而产出更健壮、更可验证的规范。

#### 7. **防止投机性功能**

模板明确抑制投机行为：

```text
- [ ] 没有投机性的、"以后可能用到"的功能
- [ ] 所有阶段都有清晰的前置条件和交付物
```

这阻止了 LLM 添加会让实现复杂化的"锦上添花"功能。每个功能都必须能追溯到一个具体的用户故事和清晰的验收标准。

### 复合效应

这些约束协同作用，产出的规范具备以下特质：

- **完整**：检查清单确保没有遗漏
- **无歧义**：强制的澄清标记凸显不确定之处
- **可测试**：测试先行的思维融入流程本身
- **可维护**：恰当的抽象层级与信息层次
- **可实现**：阶段清晰、交付物具体

模板把 LLM 从一个天马行空的写手，转变为一名有纪律的规范工程师，把它的能力导向持续产出高质量、可执行、真正驱动开发的规范。

## 宪章基石：强制执行架构纪律

SDD 的核心是一部宪章——一组治理"规范如何变成代码"的不可变原则。宪章（`memory/constitution.md`）扮演着系统"架构 DNA"的角色，确保每一次生成的实现都保持一致性、简单性和质量。

### 开发九条款

宪章定义了九个条款，塑造开发过程的方方面面：

#### 第一条：库优先原则

每个功能都必须以独立库的形式起步——没有例外。这从一开始就强制模块化设计：

```text
Specify 中的每个功能都必须（MUST）以独立库的形式开始其生命周期。
任何功能都不得直接在应用代码中实现，
必须先抽象为可复用的库组件。
```

这条原则确保规范生成的是模块化、可复用的代码，而不是铁板一块的单体应用。LLM 生成实现计划时，必须把功能组织为边界清晰、依赖最小的库。

#### 第二条：CLI 接口强制要求

每个库都必须通过命令行接口暴露其功能：

```text
所有 CLI 接口都必须（MUST）：
- 接受文本输入（通过 stdin、参数或文件）
- 产出文本输出（通过 stdout）
- 支持 JSON 格式的结构化数据交换
```

这强制保证了可观测性和可测试性。LLM 无法把功能藏进不透明的类里——一切都必须能通过基于文本的接口访问和验证。

#### 第三条：测试先行铁律

最具变革性的一条——先有测试，后有代码：

```text
这一点不容妥协（NON-NEGOTIABLE）：所有实现都必须严格遵循测试驱动开发。
在以下条件满足之前，不得编写任何实现代码：
1. 单元测试已编写
2. 测试已经用户验证和批准
3. 确认测试处于失败状态（红灯阶段）
```

这彻底反转了传统的 AI 代码生成。LLM 不能生成代码再祈祷它能跑，而必须先生成定义行为的全面测试，获得批准之后，才能生成实现。

#### 第四、五、六条：由项目自定义的治理

第四、五、六条有意留给各项目的宪章来定义，而不是由 Spec Kit 规定。宪章模板提供了占位槽位和示例关注点，比如集成测试、可观测性、版本管理和破坏性变更，团队用与自身系统和组织相匹配的原则替换这些占位内容。

这既保持了九条款结构的稳定，又允许每个项目写入自己不可妥协的标准。对某个项目来说，第四条可能治理安全与访问边界；对另一个项目，它可能定义集成测试要求。`/speckit.analyze` 命令评估的是项目中的具体宪章，因此这些由项目自定义的条款和内置示例一样，参与合规检查。

#### 第七、八条：简单性与反抽象

这对条款联手对抗过度设计：

```text
第 7.3 节：最小项目结构
- 初始实现最多 3 个项目
- 更多项目需要书面论证

第 8.1 节：信任框架
- 直接使用框架功能，而不是包一层
```

当 LLM 可能下意识地构建繁复抽象时，这些条款迫使它为每一层复杂度给出理由。实现计划模板的"阶段 -1 关卡"直接落实这些原则。

#### 第九条：集成优先测试

优先真实场景测试，而非孤立的单元测试：

```text
测试必须（MUST）使用贴近真实的环境：
- 优先使用真实数据库而非 mock
- 使用真实服务实例而非桩（stub）
- 契约测试必须在实现之前完成
```

这确保生成的代码在实践中可用，而不只是理论上成立。

### 通过模板执行宪章

实现计划模板通过具体的检查点让这些条款落地执行：

```markdown
### 阶段 -1：实现前关卡

#### 简单性关卡（第七条）

- [ ] 项目数量 ≤3 个？
- [ ] 没有为未来过度设计？

#### 反抽象关卡（第八条）

- [ ] 直接使用框架能力？
- [ ] 单一模型表示？

#### 集成优先关卡（第九条）

- [ ] 契约已定义？
- [ ] 契约测试已编写？
```

这些关卡就像架构原则的编译期检查。LLM 要么通过关卡，要么在"复杂度跟踪"小节记录有正当理由的例外，否则无法继续。

### 不可变原则的力量

宪章的力量在于它的不可变性。实现细节可以演进，核心原则却始终不变。这带来了：

1. **跨越时间的一致性**：今天生成的代码与明年生成的代码遵循同样的原则
2. **跨越 LLM 的一致性**：不同的 AI 模型产出架构上兼容的代码
3. **架构完整性**：每个功能都在强化而不是侵蚀系统设计
4. **质量保证**：测试先行、库优先、简单性原则共同确保代码可维护

### 宪章的演进

原则不可变，但其应用方式可以演进：

```text
第 4.2 节：修订流程
修改本宪章需要：
- 明确记录变更理由
- 项目维护者评审并批准
- 向后兼容性评估
```

这让方法论能够学习和改进，同时保持稳定。宪章用带日期的修订记录展示自身的演进，说明原则可以基于真实世界的经验不断打磨。

### 超越规则：一种开发哲学

宪章不只是一本规则手册——它是一种塑造 LLM 代码生成思维方式的哲学：

- **可观测胜于不透明**：一切都必须能通过 CLI 接口检视
- **简单胜于精巧**：从简单开始，只在被证明必要时增加复杂度
- **集成胜于孤立**：在真实环境中测试，而不是人造环境
- **模块胜于单体**：每个功能都是边界清晰的库

通过把这些原则嵌入规范和规划流程，SDD 确保生成的代码不只是能跑——它是可维护、可测试、架构上健全的。宪章把 AI 从代码生成器转变为尊重并强化系统设计原则的架构伙伴。

## 变革

这无关取代开发者，也无关把创造力自动化。它是通过自动化机械式的转换来放大人的能力，是构建一个紧密的反馈循环：规范、研究与代码共同演进，每一轮迭代都带来更深的理解，以及意图与实现之间更好的对齐。

软件开发需要更好的工具来维持意图与实现的对齐。SDD 提供了达成这种对齐的方法论：用能生成代码的可执行规范，而不是仅仅给代码提供指导。
